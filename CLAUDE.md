# CLAUDE.md

本文件为 Claude Code (claude.ai/code) 在此代码仓库中工作时提供指导。
`AGENTS.md` 是 `CLAUDE.md` 的软链接，本质上是同一个文件，不要重复修改两份。

## 仓库概述

这是一个专注于**基于大语言模型（LLM）的控制系统状态机建模与验证**的博士研究仓库。研究旨在构建一个覆盖"生成-验证-修复"全生命周期的自动化、迭代式闭环方法，系统性解决从非形式化需求到高可信度形式化模型的转化难题。

## 研究背景

本仓库包含博士论文的相关材料，涵盖四大研究主题：

1. **基于控制系统软件需求的LLM状态机结构化建模方法**
2. **基于模型元素的验证场景与待验证性质生成方法**
3. **基于验证剖面的状态机验证方法**
4. **面向已知缺陷的迭代式模型修复方法**

关键技术焦点：
- 时间自动机理论与时间约束
- 基于模型检查的形式化验证
- 安全性质、活性性质和时序逻辑（LTL/CTL）
- 带守卫条件的层次化状态机
- LLM能力与形式化方法的融合

## 工具使用说明

### PDF文本提取工具（tools/pdf_extractor.py）

**重要：处理PDF文件时，优先使用此工具而非其他方法**

这是一个专门为本研究项目设计的PDF文本提取工具，支持两种提取模式：

#### 使用方法

```bash
# 文字模式（快速，适用于数字化PDF）
python -m tools.pdf_extractor -i document.pdf -o output.txt -m text

# OCR模式（适用于扫描版PDF）
python -m tools.pdf_extractor -i scanned.pdf -o output.txt -m ocr
```

#### 功能特点

- **文字模式（text）**：直接提取PDF中的文本内容，速度快，适用于电子版PDF
- **OCR模式（ocr）**：使用Tesseract OCR识别图像中的文字，支持中英文混合识别（`chi_sim+eng`），适用于扫描版PDF
- 自动按页码分隔内容（`--- Page N ---`）
- 输出UTF-8编码的文本文件

#### 重要提示

**如果使用文字模式提取的结果出现显著异常（如乱码、大量缺失内容、格式严重错乱等），请立即切换到OCR模式重新提取。**

某些PDF虽然看起来是电子版，但实际上是图片格式或使用了特殊编码，导致文字模式无法正确提取。此时OCR模式虽然较慢，但能获得更准确的结果。

#### 依赖库

- 文字模式：`PyPDF2>=3.0.0`
- OCR模式：`pdf2image>=1.16.0`, `pytesseract>=0.3.10`, `Pillow>=10.0.0`

## 论文文件管理规范

论文可以出现在仓库的任何路径下（如各个project目录、专门的文献目录等）。每篇论文应按以下规范组织：

### 目录结构

每篇论文创建一个独立的子目录，目录名应简洁明了（通常使用论文的关键词或工具名），例如：
```
project_1_llm_state_machine_modeling/
├── related_work/
│   ├── ttool-ai/
│   │   ├── Apvrille和Sultan - 2024 - System Architects Are not Alone Anymore....pdf
│   │   ├── paper_content.txt
│   │   ├── desc.md
│   │   └── bibtex.bib          # 可选
│   └── another-paper/
│       ├── Author - Year - Title.pdf
│       ├── paper_content.txt
│       ├── desc.md
│       └── bibtex.bib          # 可选
```

### 文件说明

1. **PDF原文**：论文的PDF文件，文件名格式建议为 `作者 - 年份 - 标题.pdf`

2. **paper_content.txt**：使用 `tools/pdf_extractor.py` 自动提取的论文文本内容
   ```bash
   # 提取命令示例
   python -m tools.pdf_extractor -i "path/to/paper/论文.pdf" -o "path/to/paper/paper_content.txt" -m text
   ```

3. **desc.md**：论文的结构化总结文档（**核心文件**）

4. **bibtex.bib**（可选）：论文的BibTeX引用信息
   - 包含论文的完整引用格式
   - 便于快速生成带参考文献的内容
   - 可用于LaTeX文档、学术写作等场景

### desc.md 编写规范

`desc.md` 是对论文的深度分析和总结，应包含以下内容：

#### 必需章节

1. **论文基本信息**
   - 标题（中英文）
   - 作者及单位
   - 发表年份和会议/期刊
   - DOI和论文链接（如有）

   **新增必需信息（获取方式）**：
   - **代码/仓库获取方式**：
     - 如果论文开源了代码、仓库或实验工件，必须在 `desc.md` 中明确写出可访问链接（如 GitHub、项目主页、Artifact 页面）
     - 如果论文未提供公开链接，必须明确写出“原文未提供公开代码/仓库获取链接”，并说明当前可行获取方式（如联系作者、关注后续发布）
   - **数据集获取方式**：
     - 如果论文使用或发布了数据集，必须在 `desc.md` 中明确写出数据集获取链接或获取路径（如下载页、仓库目录、补充材料）
     - 如果原文未提供可访问链接，必须明确写出“原文未提供公开数据集获取链接”，并说明当前可行获取方式
   - **注意**：这里强调的是“如何获取（access/retrieval）”，不是“如何制作（construction）”。

2. **简报**（**新增必需章节**）

   简报部分应在基本信息之后、研究问题与动机之前，用简明扼要的方式回答以下五个问题：

   **a. 解决的问题**
   - 明确说明论文解决了什么问题或证实了什么结论
   - **必须包含**：输入是什么、方法是什么、输出是什么
   - 每个部分（输入/方法/输出）可能包含多个元素，不能缺少任何一个
   - 在文字说明后，**必须配上字符示意图**，展示从输入层→方法层→输出层的完整流程
   - 示意图应清晰展示黑箱视角下的系统结构
   - 最后用一句话总结主要实验结果或性能提升数据

   **b. 研究动机**
   - 出于什么原因需要做这项研究
   - 前人的工作差在哪里
   - 发现了什么问题或局限性

   **c. 方法创新**
   - 提出的方法是什么
   - 核心创新点有哪些（通常2-4个）
   - 每个创新点用一句话说明

   **d. 实验设计**
   - 实验是怎么做的
   - 对比对象包含谁（baseline是什么）
   - 使用了什么数据集
   - 评估指标是什么
   - 使用的模型或工具

   **e. 结论与不足**
   - 得到了什么主要结论
   - 方法的优势在哪里
   - 存在什么局限性或不足（通常3-5个）

   每个问题用一小段话（50-150字）讲明白，保持简洁清晰。

3. **研究问题与动机**（详细版）
   - 论文要解决什么问题？
   - 为什么这个问题重要？
   - 现有方法的局限性是什么？

4. **核心方法**
   - 论文提出的主要方法/技术/框架
   - 关键技术细节和创新点
   - 方法的形式化定义（如有）
   - 系统架构或流程图描述

5. **实验与评估**
   - 使用的数据集和评估指标
   - 主要实验结果
   - 与基线方法的对比
   - 方法的优势和局限性

6. **与本研究的关系**
   - 该工作如何与本博士研究相关？
   - 可以借鉴的思路或技术
   - 存在的不足和改进空间
   - 对本研究的启发

7. **重要的相关工作**（**核心章节**）

   这是desc.md中最重要的章节之一，需要系统性地整理论文中实际引用的重要文献。**必须从论文原文中提取**，不能编造或臆测。

   **编写流程**：
   1. 仔细阅读论文的"相关工作"章节（Related Work）
   2. 阅读论文的参考文献列表（References）
   3. 在论文正文中查找每篇文献的引用位置和引用内容
   4. 根据每篇文献在论文中的作用进行分类

   **文献分类标准**（按以下五类组织）：

   **第一类：重要的前身类工作**
   - 定义：与本论文一脉相承的前期工作，可以是作者团队自己的前期工作，也可以是其他团队的工作，只要本论文明确基于该工作进行改进或继续发展
   - 识别标准：
     - 作者团队自己的工作：论文中明确说"our previous work"、"we previously"、"in our earlier work"等
     - 其他团队的工作：论文中明确说"building on [X]"、"based on [X]"、"extending [X]"、"improving upon [X]"、"inspired by [X]"等，表明本工作是在该工作基础上发展而来
     - 论文在介绍中或相关工作中明确指出本工作是对该工作的直接改进或扩展
   - 作用：展示研究的演进脉络，说明本工作是如何从前期工作发展而来的
   - 示例：
     - MIG论文中的Chen et al. (2023)单步领域建模工作（作者团队自己的前期工作）
     - 如果某论文明确说"We build upon the approach proposed by [X] and extend it to..."，则[X]也属于前身工作

   **第二类：直接参与实验的baseline**
   - 定义：在实验中作为对比基准的工作
   - 识别标准：
     - 论文实验部分明确与之对比
     - 有具体的性能数据对比（如表格、图表中出现）
     - 论文中说"compared with"、"baseline"、"we evaluate against"等
   - 作用：提供实验对比基准，证明本方法的有效性
   - 注意：前身工作可能同时也是baseline（如MIG的情况）

   **第三类：提供了重要论证的工作**
   - 定义：为本论文的研究动机、问题定义、方法设计提供了论证支持
   - 识别标准：
     - 在相关工作中被详细讨论
     - 论文指出其局限性或不足
     - 用于论证本研究的必要性
     - 论文中说"however"、"but"、"lacks"、"limited"等表示不足的词汇
   - 作用：论证本研究的必要性和创新性
   - 示例：
     - 揭示现有方法的局限性（如Cámara et al.指出ChatGPT在建模任务中的不足）
     - 代表传统方法的不足（如基于规则、统计方法的局限）

   **第四类：在技术上提供了支持的工作**
   - 定义：本论文直接使用、采用或受启发的技术、框架、方法
   - 识别标准：
     - 论文中说"we adopt"、"we use"、"building on"、"inspired by"、"based on"等
     - 论文方法章节中明确提到使用了该技术
     - 是本工作的技术组成部分
   - 作用：说明本工作的技术基础和来源
   - 示例：
     - Few-shot learning（MIG直接采用）
     - Chain-of-Thought（MIG受启发进行任务分解）
     - GPT-4（MIG直接使用的模型）
     - LangChain（MIG使用的框架）
   - **重要区分**：
     - 第三类（论证）vs 第四类（技术使用）：前者是"指出问题"，后者是"提供解决方案"
     - 第四类（技术使用）vs 第五类（其他）：前者是"直接采用"，后者是"提供参考"

   **第五类：其他重要工作**
   - 定义：提供背景知识、理论基础、评估方法等支撑性内容
   - 识别标准：
     - 不直接参与本工作的设计或实验
     - 提供评估指标、理论基础、背景知识
     - 论文中引用但未详细讨论
   - 作用：为本研究提供支撑性知识
   - 示例：
     - 评估指标来源（如Bian et al.的类图评分方法）
     - LLM综述（提供背景知识）
     - 教材和理论基础（如UML标准、软件工程理论）

   **每篇文献必须包含的信息**：

   a. **基本信息**
   - 作者和年份
   - 完整标题
   - 发表会议/期刊
   - 如果是作者团队的前期工作，需明确标注

   b. **主要内容**
   - 详细说明论文研究了什么
   - 使用了什么方法
   - 方法的特点和创新点

   c. **论文中的引用**（**关键信息**）
   - 在论文中的具体引用位置（如"第XXX行"）
   - 论文中如何引用它的（引用的具体内容）
   - 论文对该工作的评价或讨论
   - 示例：
     ```
     论文中的引用：
     - 第736-739行："Few-shot prompting, proposed by Brown et al., introduces..."
     - 第371-373行："We adopt N-shot learning in our prompt design"
     ```

   d. **主要发现/局限性**（如适用）
   - 该工作的主要发现或结论
   - 论文指出的局限性或不足
   - 实验结果或性能数据

   e. **与本论文的关系**（**核心内容**）
   - 详细说明该工作如何与本论文相关
   - 本论文如何改进、借鉴或对比该工作
   - 该工作为本论文提供了什么支撑
   - 根据分类说明具体作用：
     - 第一类：说明研究演进关系
     - 第二类：说明实验对比情况和性能提升数据
     - 第三类：说明论证了什么问题或必要性
     - 第四类：说明技术如何被使用或启发了什么设计
     - 第五类：说明提供了什么支撑性知识

   f. **佐证内容/技术使用/作用**（根据分类）
   - 第一类：为本研究的演进提供了什么基础
   - 第二类：提供了什么实验对比数据
   - 第三类：论证了什么问题或必要性
   - 第四类：技术如何被使用（直接采用/启发设计/提供框架）
   - 第五类：提供了什么支撑（评估方法/背景知识/理论基础）

   **编写注意事项**：

   1. **必须基于原文**：所有信息必须从论文原文中提取，不能编造或臆测
   2. **引用位置必须准确**：标注论文中的具体行号或章节
   3. **分类必须准确**：根据文献在论文中的实际作用进行分类，不能主观臆断
   4. **避免重复**：同一篇文献可能属于多个类别（如前身工作同时也是baseline），在主要类别中详细描述，在其他类别中简要说明并引用主要描述
   5. **保持客观**：如实反映论文对该工作的评价，不添加个人观点
   6. **详细程度**：
      - 第一、二、三、四类：详细描述（每篇200-400字）
      - 第五类：简要描述（每篇100-200字）

### 工作流程

处理新论文时，建议按以下步骤操作：

1. 在合适的位置（如相关project目录下）创建新的论文目录
2. 将PDF文件放入目录
3. 使用 `tools/pdf_extractor.py` 提取文本到 `paper_content.txt`
4. 阅读论文和提取的文本
5. 按照规范编写 `desc.md`
6. （可选）创建 `bibtex.bib` 文件，包含论文的BibTeX引用信息
7. 必要时更新 `TARGET.md` 中的文献综述部分

### 阅读现有论文的推荐顺序

**重要**：当需要了解一篇现存论文的内容时，应按以下顺序阅读：

1. **首先阅读 `desc.md`**：这是论文的结构化总结，包含简报、核心方法、实验结果和与本研究的关系分析，能快速了解论文的核心内容
2. **如需更多细节，再阅读 `paper_content.txt`**：这是从PDF提取的完整文本，包含所有技术细节、公式和实验数据

这种顺序能够：
- 快速把握论文的核心思想和贡献
- 避免在大量原始文本中迷失
- 根据需要逐步深入了解细节
- 节省时间和精力

### 生成desc.md时使用bibtex.bib

**重要**：在生成 `desc.md` 时，如果同路径下已存在 `bibtex.bib` 文件，应该：

1. **读取 `bibtex.bib` 文件**：获取论文的完整引用信息
2. **将BibTeX信息纳入基本信息章节**：
   - 从BibTeX中提取标题、作者、会议/期刊、年份、DOI等信息
   - 确保基本信息的准确性和完整性
   - 避免手动输入可能导致的错误

这样可以：
- 保证基本信息的准确性
- 节省手动查找和输入的时间
- 确保引用格式的标准化

### 提取控制系统状态机自然语言内容时的 STM 规范

当任务是从给定文献资料中提取“带有状态机要素的控制系统设计/描述/需求等自然语言内容”时，必须遵循：

- `project_1_llm_state_machine_modeling/STM_GUIDE.md`

执行这类任务时，至少满足以下要求：

1. 只提取**真实控制系统**、控制器或控制子系统的设计/功能/需求描述，不能把开发流程、分析流程、工具模式、需求管理状态等内容误当成控制系统状态机样本。
2. 所有写入 `STM.md` 的内容都必须可追溯到论文原文；证据优先来自 `paper_content.txt`，必要时回到 PDF 原文核对。
3. 若 `paper_content.txt` 提取质量差，应优先使用 `tools/pdf_extractor.py` 重新提取，必要时切换到 OCR 模式，再进行提取。
4. `STM.md` 的章节结构、盘点逻辑、条目判定、原文摘录方式、整理后的自然语言描述写法、逐句溯源方式，以及“未收获”场景的记录方式，均以 `STM_GUIDE.md` 为准。
5. 整理后的自然语言内容应尽量沿用原文措辞，只允许做必要转接；不得编造任何在原文中找不到依据的内容。

## 核心技术概念

### 状态机形式化
- 状态机模型：`M = (S, E, V, Tr, A)` 其中 S=状态集合，E=事件集合，V=变量集合，Tr=迁移集合，A=动作集合
- 时间状态机：`TSM = (S, S₀, E, V, C, Tr, Inv, Act)` 包含时钟约束和不变式
- 验证剖面：`SP = ⟨(E₁, C₁, τ₁), (E₂, C₂, τ₂), ..., (Eₙ, Cₙ, τₙ)⟩`
- 性质模型：`Property = (Type, Scope, Predicate, TemporalConstraint)`

### 缺陷类型
- δ型：迁移守卫错误
- τ型：时间约束违反
- state型：状态缺失或冗余

### 相关工具与框架
- UPPAAL：时间自动机验证工具
- pyfcstm：状态机领域特定语言（https://github.com/HansBug/pyfcstm）
- UML/SysML：状态机建模标准
