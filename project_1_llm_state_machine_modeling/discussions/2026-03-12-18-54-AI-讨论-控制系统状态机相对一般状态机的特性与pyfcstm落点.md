# 控制系统状态机相对一般状态机的特性、痛点对应与 pyfcstm 落点

## 1. 这份讨论稿要回答什么

这份讨论稿聚焦三个问题：

1. 我们的【控制系统状态机】相对一般主流状态机，究竟多强调了哪些额外特性。
2. 这些额外特性分别在解决控制系统建模中的哪些真实痛点。
3. 在不深扩 `pyfcstm` 的前提下，我们现在能把这件事做到什么程度。

这里先给一个总判断：

> 我们不是要另造一种完全脱离 UML/SysML/Harel statechart 的新状态机，而是要把控制系统里真正关键、真正反复出问题的那一层语义，稳定地提升为建模、仿真、验证、修复都可复用的工作对象。

因此，所谓“控制系统状态机”的额外性，重点不在于比一般状态机再多发明几条语法，而在于它必须把控制系统最关键的语义用得更稳定、更明确、更适合验证闭环。

## 2. 一个前提：普通状态机不是完全不能做，而是做得不够好

需要先避免一个说法陷阱。

普通状态机并不是“绝对画不出”控制系统逻辑。很多控制系统逻辑，理论上都可以用 UML/SysML 状态机、一般 Harel statechart 甚至普通 FSM 图画出来。

问题在于，普通状态机通常存在下面几个不足：

1. 它对控制系统最关键的语义没有提升为一等关注对象，导致这些语义经常散落在文字说明、备注和隐含约定里。
2. 它更强调“可表示”，而不是“可稳定抽取、可稳定执行、可稳定验证、可稳定修复”。
3. 它的泛化能力很强，但也因此缺少控制系统领域里真正重要的角色划分，比如模式、联锁、故障恢复、事件来源、控制权限、安全退化等。
4. 不同工具和建模者对层次语义、生命周期语义、守卫优先级、异常路径的处理方式不够统一，导致后续仿真、验证和代码生成容易漂移。

所以，我们要论证的不是“普通状态机完全不能用”，而是：

> 对控制系统而言，普通状态机缺少一套稳定的控制语义组织方式；  
> 我们的【控制系统状态机】要解决的是这个问题，而 `pyfcstm` 是当前可落地的承载内核。

## 3. 我认为最重要的 6 类额外特性

下面这 6 类特性，是我认为最适合拿来定义【控制系统状态机】的核心差异项。  
每一类下面都回答四件事：

1. 现有论文集中有哪些现成例子。
2. 普通状态机为什么做不好，或者至少做得不够好。
3. 我们的控制系统状态机需要额外做到什么。
4. 基于 `pyfcstm` 这副牌，现在能落到什么程度。

### 3.1 模式化与层次化的控制组织

#### 现有论文集里的典型例子

1. [Bell 412 自主飞行 supervisor 的 Takeoff / On Route / Landing 分解](../sources/methodology-to-develop-a-discrete-event-supervisory-controller-for-an-autonomous-helicopter-flight/STM.md)
2. [包装机械模块的 Automatic / Setup / Manual / Semi-automatic / Initialize / Shutdown / Safe stop 模式组织](../sources/fault-handling-plc-industry4/STM.md)

这些例子共同说明：控制系统不是简单的“状态列表”，而是天然带有监督层与执行层、模式层与局部动作层。

#### 普通状态机为什么做不好或做不了

普通状态机当然可以把这些状态全画出来，但常见问题是：

1. 它没有强迫建模者区分“全局模式状态”和“局部执行状态”，结果经常把模式切换和局部动作揉成同一层扁平状态表。
2. 在需求抽取阶段，模式、阶段、子动作的角色不清楚，LLM 和人工都容易把“任务阶段”误写成“叶子动作状态”。
3. 一旦把监督层和执行层混平，后续的验证目标也会变得模糊，因为你很难单独讨论“模式切换是否合法”和“模式内部动作是否完整”。

所以，不是普通状态机完全不能画，而是它没有把“模式组织”当成控制系统的一等建模问题。

#### 我们的控制系统状态机需要额外做到什么

这里我们要额外强调的是：

1. 明确区分运行模式、任务阶段、局部动作子状态。
2. 允许上层 supervisor 管模式和控制权限，下层子状态机管执行细节。
3. 把“模式切换条件”和“模式内部执行逻辑”分开看、分开验证、分开修复。

这会直接带来两个额外收益：

1. 从自然语言需求中抽取状态机时，更容易把复杂控制逻辑组织成稳定结构。
2. 在验证与修复时，可以明确定位问题究竟出在模式切换层，还是模式内部执行层。

#### 基于 pyfcstm 现在能做到什么

这件事基本已经能直接落在 `pyfcstm` 上，而不需要深扩语言。

现有支撑点包括：

1. [层次状态机和复合状态能力](https://github.com/HansBug/pyfcstm/blob/main/README.md)
2. [DSL 中的嵌套状态、初始转换和状态层次表达](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/dsl/index_zh.rst)
3. [仿真语义里对叶状态、复合状态和可停止状态的区分](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/simulation/index_zh.rst)

因此，基于 `pyfcstm`，我们现在完全可以把【控制系统状态机】先收敛为：

> 面向离散监督控制层的层次化状态机。

这比普通状态机更强的地方，不是语法更花，而是我们能更稳定地表达“模式层 + 执行层”的组织关系。

### 3.2 守卫化的联锁与许可控制

#### 现有论文集里的典型例子

1. [铁路/有轨电车联锁中的 route reservation、冲突进路互斥、道岔定位与信号开放](../sources/a-formal-approach-for-the-construction-and-verification-of-railway-contr/STM.md)
2. [SSI 联锁中的 request-check-lock-green-release route lifecycle](../sources/verification-of-railway-interlocking-systems/STM.md)
3. [电梯中的超载禁止起动、门口有障碍时禁止关门](../sources/design-amp-control-of-an-elevator-control-system-using-plc/STM.md)

这些样本都不是“事件一来就跳转”，而是“事件到了，且许可条件满足，才允许切换”。

#### 普通状态机为什么做不好或做不了

普通状态机最大的问题不是没有 guard，而是 guard 太“泛”了。

具体说：

1. 一般状态机里的 guard 只是一个通用条件语法，没有把它当成联锁、许可、禁止、资源占用这类控制语义来显式看待。
2. 结果很多真实的 interlock 条件会散落在备注、补充说明、外部约束表里，而不是稳定落在迁移上。
3. 一旦 guard 缺失，模型表面上仍然“能跑”，但安全逻辑已经坏掉了；普通状态机很难把这类问题显式当成一类高优先级缺陷。
4. 对 LLM 建模来说，普通状态机也没有告诉模型哪些自然语言条件应优先识别成“联锁 guard”，哪些只是普通业务前置条件。

所以，普通状态机不是不能表达 guard，而是不足以把 guard 当成控制系统最关键的语义骨架。

#### 我们的控制系统状态机需要额外做到什么

这里要提升为一等对象的，不是“条件”本身，而是“控制许可条件”。

具体应强调：

1. 哪些 guard 属于安全联锁。
2. 哪些 guard 属于使能条件。
3. 哪些 guard 属于禁止条件或资源互斥条件。
4. 哪些 guard 缺失会直接变成高风险缺陷。

这会让我们能比普通状态机多做两件事：

1. 在需求到模型阶段，更稳定地把自然语言中的控制约束落成 guard。
2. 在验证与修复阶段，把“guard 缺失 / guard 错误”作为明确的缺陷类别处理。

#### 基于 pyfcstm 现在能做到什么

这件事 `pyfcstm` 也已经具备核心承载能力：

1. [DSL 原生支持 transition guard 和表达式系统](https://github.com/HansBug/pyfcstm/blob/main/README.md)
2. [DSL 教程中给出了条件、变量、表达式和迁移的组合方式](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/dsl/index_zh.rst)
3. [仿真语义中支持基于迁移定义顺序的优先级处理](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/simulation/index_zh.rst)

因此，我们不需要先发明“联锁迁移”这种全新语法。  
更现实的做法是：把 `pyfcstm` 的 guard 解释成控制 guard，并在研究流程里把它们分类为联锁、许可、禁止、恢复条件。

这一步比普通状态机更进一步，因为我们不是泛泛地“有 guard”，而是明确要求它们承担控制约束的主体角色。

### 3.3 异常、降级与恢复链

#### 现有论文集里的典型例子

1. [包装机械中的 aborting -> aborted -> resetting -> starting -> automatic 恢复链](../sources/fault-handling-plc-industry4/STM.md)
2. [CARA 输液泵在故障或无效血压条件下释放自动控制并退回 manual mode](../sources/cara-infusion-pump-formal-spec/STM.md)
3. [电梯在供电故障时就近平层并开门的保护动作](../sources/design-amp-control-of-an-elevator-control-system-using-plc/STM.md)

这些例子都说明：对控制系统而言，异常路径不是边角逻辑，而是主逻辑的一部分。

#### 普通状态机为什么做不好或做不了

普通状态机通常把异常处理做成一个宽泛的 `Error` 状态，问题在于这远远不够。

控制系统真正关心的是：

1. 故障从哪里触发。
2. 触发后进入的是故障态、降级态、人工接管态，还是安全停机态。
3. 恢复是自动完成、人工确认、重新标定，还是根本不允许自动恢复。
4. 何时允许重新回到 nominal path。

一般状态机缺少的不是“Error 状态”，而是“异常链条的角色语义”。  
如果这一层没有被稳定显式化，那么：

1. LLM 生成时容易只保留 nominal path，漏掉 recovery chain。
2. 验证时很难聚焦“fault 后是否安全退化”和“恢复条件是否过宽”。
3. 修复时也很难知道要补的是 fault entry、fallback path，还是 return condition。

#### 我们的控制系统状态机需要额外做到什么

这里我们要额外强调：

1. fault、degraded、manual takeover、safe state、recovery 这些状态角色必须清楚。
2. 异常触发路径和恢复返回路径都必须显式建模。
3. 正常路径和异常路径要同样可抽取、可仿真、可验证、可修复。

一旦做到这点，我们就比普通状态机多了一层很关键的能力：

1. 能系统性地讨论“故障后去哪”和“何时允许回来”。
2. 能把很多真实工业控制里的保护逻辑纳入模型主体，而不是留在自然语言说明里。

#### 基于 pyfcstm 现在能做到什么

这件事同样可以在不深扩 `pyfcstm` 的前提下落地：

1. [普通故障迁移、恢复迁移和生命周期动作都可直接建模](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/dsl/index_zh.rst)
2. [DSL 提供强制转换语法，适合表达统一故障跳转或保护跳转](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/dsl/index_zh.rst)
3. [仿真语义能承载 enter / during / exit 中的 reset、cleanup、acknowledge 等动作](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/simulation/index_zh.rst)

因此，基于 `pyfcstm`，我们现在完全可以做一类“异常和恢复链显式化”的控制系统状态机。  
这已经明显强于普通状态机的宽泛 `Error` 画法。

### 3.4 分层事件与控制权限作用域

#### 现有论文集里的典型例子

1. [PLC 电梯控制器中对 operational-mode、safety signal、car-call、hall-call、floor sensor、door signal 等输入的区分](../sources/design-of-a-plc-based-elevator-control-system/STM.md)
2. [Bell 412 supervisor 同时接收 FCC、mission manager、pilot、aircraft 等多来源输入](../sources/methodology-to-develop-a-discrete-event-supervisory-controller-for-an-autonomous-helicopter-flight/STM.md)
3. [CARA 中 caregiver、pump、blood pressure sensor、algorithm 各自承担不同控制交互角色](../sources/cara-infusion-pump-formal-spec/STM.md)

这些例子都表明，控制系统里的事件不是一堆同质名字，而是带来源、带权限、带传播范围的。

#### 普通状态机为什么做不好或做不了

普通状态机通常把事件处理成“名字 + 触发”，这对控制系统来说不够。

不够的地方在于：

1. 它很少显式区分命令事件、传感事件、故障事件、内部控制事件、全局急停事件。
2. 它也很少约束一个事件应该在哪个层级可见、由谁触发、影响到哪一层状态。
3. 结果就是事件语义容易扁平化，同名不同义、局部事件误建成全局事件、全局保护事件又被错误局部化。

这在控制系统里不是小问题，因为事件来源本身就意味着控制权限。  
谁能触发，在哪一层触发，决定了模型是否忠实于真实控制结构。

#### 我们的控制系统状态机需要额外做到什么

这里要额外强调的是事件角色，而不是只强调事件存在。

至少应明确：

1. 哪些是操作员/维护者命令。
2. 哪些是传感/检测输入。
3. 哪些是控制器内部事件。
4. 哪些是故障/报警事件。
5. 哪些是全局广播或跨层共享事件。

这样一来，我们就比普通状态机多出两类能力：

1. 更容易把自然语言中的“谁发的、给谁的、影响哪一层”的信息抽进模型。
2. 更容易在验证时讨论“控制权限是否越界”“局部事件是否被错误放大为全局事件”。

#### 基于 pyfcstm 现在能做到什么

这是 `pyfcstm` 很有辨识度的一张牌。

它已经支持：

1. [局部事件 `::`](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/dsl/index_zh.rst)
2. [链式事件 `:`](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/dsl/index_zh.rst)
3. [绝对事件 `/`](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/dsl/index_zh.rst)
4. [仿真运行时对事件触发和可用事件的处理](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/simulation/index_zh.rst)

这意味着我们不需要先扩一个复杂的 port/interface/event bus 元模型，就已经能把控制系统中的局部命令、上层命令、全局保护事件做出相当清楚的分层。

这比普通状态机更进一步，因为普通状态机往往只有“事件名”，而 `pyfcstm` 已经给了我们“事件作用域”这张可直接落地的牌。

### 3.5 明确、可执行的生命周期语义

#### 现有论文集里的典型例子

1. [洗衣机控制中的 fill -> wash forward/stop/reverse -> drain -> rinse -> dehydrate 顺序链](../sources/automatic-washing-machine-control-system-based-on-plc/STM.md)
2. [SSI 联锁中的 request -> check -> command/lock -> green -> release 生命周期](../sources/verification-of-railway-interlocking-systems/STM.md)
3. [电梯在无请求时保持停层开门、收到请求后执行门控与运行逻辑](../sources/design-of-a-plc-based-elevator-control-system/STM.md)

这些例子都强调一件事：控制系统状态机不是“画图工具里的图”，而是必须能回答执行顺序。

#### 普通状态机为什么做不好或做不了

普通状态机在工程落地时，经常败在语义不够硬。

典型问题包括：

1. `enter / during / exit` 的先后次序不够明确，或者不同工具处理不一致。
2. 父状态和子状态的动作谁先谁后，经常依赖工具默认值或隐含约定。
3. 图上看起来差不多，仿真、验证、代码生成却可能语义漂移。

对普通业务流程，这种漂移可能只是实现细节；  
对控制系统，这种漂移本身就可能改变控制逻辑，因为动作顺序就是控制逻辑的一部分。

#### 我们的控制系统状态机需要额外做到什么

这里我们真正要强调的是：

1. 进入、驻留、退出动作必须有明确语义。
2. 父子状态边界上的动作必须有明确执行规则。
3. 这个语义要足够硬，能被仿真、验证、生成等后续环节复用。

一旦做到这一点，我们就能比普通状态机多做三件事：

1. 把模型当成可执行模型，而不是示意图。
2. 稳定做语义一致的仿真与对拍。
3. 把缺陷定位到具体的生命周期动作和转换链，而不是只停留在“图有问题”。

#### 基于 pyfcstm 现在能做到什么

这也是 `pyfcstm` 当前最有价值的支撑点之一。

它已经明确支持：

1. [`enter / during / exit` 生命周期动作](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/simulation/index_zh.rst)
2. [`>> during before/after` 这种祖先切面语义](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/simulation/index_zh.rst)
3. [复合状态边界上的 `during before/after` 语义](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/simulation/index_zh.rst)
4. [仿真运行时、热启动、状态路径控制等执行能力](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/simulation/index_zh.rst)

因此，基于 `pyfcstm`，我们不是只能“画一个控制状态机”，而是已经能把它当成一个有明确执行语义的控制状态机来运行和检查。

这比普通状态机强的地方很明确：  
我们能把“控制图”推进成“可执行控制模型”。

### 3.6 面向验证的建模友好性

#### 现有论文集里的典型例子

1. [SSI 联锁模型把 route lifecycle 组织成可自动检查安全性质的结构](../sources/verification-of-railway-interlocking-systems/STM.md)
2. [铁路控制系统把 route reservation、冲突互斥、信号开放与释放组织成可验证链路](../sources/a-formal-approach-for-the-construction-and-verification-of-railway-contr/STM.md)
3. [CARA 中“3 分钟内拿不到有效血压则退回 manual mode”这类约束天然面向验证](../sources/cara-infusion-pump-formal-spec/STM.md)

这些样本提醒我们：控制系统状态机不能只为了“能表达”，还必须为了“能验证”。

#### 普通状态机为什么做不好或做不了

普通状态机的常见短板是，它们大多是建模友好的，却不一定验证友好。

问题主要在这里：

1. 模型结构没有按验证目标组织，导致后续性质生成和场景生成很难稳定进行。
2. 模式、联锁、故障、恢复、权限这些关键信息没有规范化沉到模型元素上，验证时只能重新猜。
3. 发现问题后，很难稳定追到“缺的是状态、错的是 guard、还是异常链漏了”。

所以，普通状态机最大的问题不是不能送去验证，而是它通常不是为验证闭环而建的。

#### 我们的控制系统状态机需要额外做到什么

这里我们真正想要的是一种“验证友好结构”：

1. 模式、guard、故障链、事件层级这些信息都能被后续验证环节稳定消费。
2. 模型缺陷能回溯到具体元素，而不是只得到一个全局性的“性质不满足”。
3. 模型既能支撑仿真，也能支撑验证场景生成、性质生成和缺陷修复。

这一点直接对应你的博士课题主线。  
也就是说，我们的控制系统状态机不是单纯为了建模，而是为了支撑“生成-验证-修复”闭环。

#### 基于 pyfcstm 现在能做到什么

`pyfcstm` 现在虽然还不是一个完整的工业验证平台，但已经有足够有价值的内核基础：

1. [仿真运行时可以支撑可执行语义检查](https://github.com/HansBug/pyfcstm/blob/main/docs/source/tutorials/simulation/index_zh.rst)
2. [验证设计文档已经把状态机验证建模为基于 Z3 的符号执行与 BFS 搜索问题](https://github.com/HansBug/pyfcstm/blob/main/pyfcstm/verify/DESIGN.md)
3. [验证设计强调与 simulate 模块共享状态机语义，而不是另起一套不一致语义](https://github.com/HansBug/pyfcstm/blob/main/pyfcstm/verify/DESIGN.md)

因此，基于 `pyfcstm`，我们现在最现实、最有把握的说法不是“已经拥有完整控制系统验证体系”，而是：

> 我们已经有一个能够支撑控制系统状态机执行语义、仿真语义和验证语义逐步贯通的内核。

这比普通状态机更进一步，因为普通状态机往往止步于“建模表示”，而我们可以把模型往验证闭环推进。

## 4. 归纳：为什么需要“我们的控制系统状态机”

把上面 6 类特性合起来看，真正的结论其实很清楚。

我们需要【控制系统状态机】，不是因为普通状态机一无是处，而是因为普通状态机没有把下面这些问题稳定收进模型主体：

1. 模式与阶段怎么组织。
2. 联锁与许可条件怎么挂到迁移上。
3. 故障、降级、人工接管和恢复链怎么显式化。
4. 事件来源、作用域和控制权限怎么区分。
5. 生命周期语义如何变成可执行语义。
6. 模型如何天然服务验证、缺陷定位和后续修复。

如果没有这层“控制语义强化”，那么状态机很容易退化成：

1. 只能展示大概流程的图。
2. 靠备注补关键安全约束的图。
3. 能建模但难验证的图。
4. 能验证但很难回溯缺陷来源的图。

而我们的【控制系统状态机】要做到的，是把这些控制系统最关键的语义从“图外说明”拉回“模型主体”。

## 5. 基于 pyfcstm 这副牌，我们现在能合理宣称什么

在不深扩 `pyfcstm` 的前提下，我建议我们把可宣称范围收敛为：

> 一类面向离散监督控制层的层次化控制系统状态机。

它当前特别适合：

1. 运行模式和阶段切换明确的控制器。
2. 依赖 guard、联锁、许可条件的安全控制逻辑。
3. 故障、降级、手动接管、恢复链明确的设备控制逻辑。
4. 需要事件分层和控制权限区分的 supervisor 逻辑。
5. 需要明确执行语义并希望进入仿真/验证闭环的状态机。

它当前不宜过度宣称为：

1. 完整 hybrid automata 或连续控制律建模语言。
2. 原生完备时间自动机语言。
3. 大而全的工业标准状态机超集。
4. 把 orthogonal regions、history states、复杂端口体系都一次性吃下的统一大元模型。

这个收敛非常重要，因为它让我们的说法同时满足三件事：

1. 对一般状态机形成清楚、可论证的差异。
2. 与现有论文样本和数据建设边界一致。
3. 与 `pyfcstm` 当前真实能力对齐，不会把讨论推向“先重写 DSL”。

## 6. 一个适合当前阶段的定义建议

基于上面的讨论，我建议当前阶段把【控制系统状态机】定义为：

> 一类面向离散监督控制层的层次化状态机。  
> 它相对一般状态机，额外强调模式化组织、联锁守卫、异常恢复、事件作用域、明确执行语义和验证友好性；  
> 在工程实现上，它以 `pyfcstm` 为状态机内核，通过既有的层次状态、guard、生命周期动作、强制转换、事件作用域、仿真与验证机制来承载这些控制语义。

这一定义的优点是：

1. 它说清楚了我们和一般状态机真正差在哪里。
2. 它把“为什么需要这种状态机”落在了控制系统痛点上。
3. 它没有把研究成功前提绑在对 `pyfcstm` 的深度扩展上。

## 7. 一句话结论

最凝练的说法可以写成：

> 我们的【控制系统状态机】不是单纯“比一般状态机多几个语法点”的状态机，而是把模式组织、联锁约束、异常恢复、事件权限、执行语义和验证闭环提升为核心目标的一类状态机；在当前阶段，这件事完全可以以 `pyfcstm` 为内核，在不深扩 DSL 的前提下先落到离散监督控制层。
