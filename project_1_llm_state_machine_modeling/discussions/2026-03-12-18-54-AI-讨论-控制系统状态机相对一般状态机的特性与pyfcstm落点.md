# 控制系统状态机相对一般状态机的特性与 pyfcstm 落点

## 1. 这份讨论稿的目标

这份讨论稿不再泛泛讨论“控制系统定义”本身，而是集中回答下面三个更关键的问题：

1. 我们说的【控制系统状态机】相比现有主流【状态机】到底多了哪些额外特性。
2. 这些额外特性分别是在解决控制系统开发中的哪些典型痛点。
3. 在我们已经有 `pyfcstm` 这副牌的前提下，我们现在就能支撑出什么样的【控制系统状态机】，而不是把讨论建立在对 `pyfcstm` 做大幅扩展的前提上。

本文的基本立场是：

> `pyfcstm` 不是未来某个理想化控制系统状态机语言的“草稿”，而是我们当前已经可用的状态机内核。  
> 因此，我们要定义的【控制系统状态机】必须尽量建立在 `pyfcstm` 已有能力之上，通过建模口径、元素角色解释和研究流程组织，把它变成面向控制系统的状态机解决方案。

## 2. 一个总判断

我建议把本课题中的【控制系统状态机】理解成：

> 在一般层次化状态机基础上，进一步强调模式控制、联锁约束、异常恢复、事件分层、执行语义清晰性和可验证性的一类状态机。

这里的重点不是“它必须比 UML/SysML 多出很多全新的语法”，而是：

1. 对控制系统真正重要的那些语义必须被稳定建模。
2. 这些语义必须能服务自然语言建模、仿真、验证、修复整个研究链条。
3. 它们最好尽可能由 `pyfcstm` 现有机制直接承载，或者用轻量建模约定承载，而不是先做一个很重的新元模型。

所以，讨论的核心不应是“我们还能再往状态机里塞多少概念”，而应是“在 `pyfcstm` 现有能力下，我们能把控制系统里最关键、最痛的那一层问题吃下来多少”。

## 3. 一般状态机和控制系统状态机的差别，不应放在语法层，而应放在任务层

如果只从教科书上看，一般状态机已经有：

1. 状态。
2. 事件。
3. 转换。
4. 条件。
5. 动作。
6. 有时还支持层次、复合状态、伪状态。

所以控制系统状态机相对一般状态机的“额外性”，不应该理解成“发明一个全新的第六元、第七元、第八元”，而应理解成下面几类偏向：

1. **状态的意义更偏运行模式和控制阶段，而不是任意业务阶段。**
2. **转换的意义更偏联锁、许可、切换、恢复和保护，而不是普通流程跳转。**
3. **事件更偏命令、传感、故障、超时、内部控制信号，而不是泛业务消息。**
4. **动作更偏控制输出、模式初始化、复位、监控和保护，而不是普通副作用。**
5. **模型必须更强调执行语义和验证友好性，而不是只要能画图。**

换句话说，控制系统状态机不是“另一种语法的状态机”，而是“**问题导向更强、执行语义更硬、验证目标更明确的状态机**”。

## 4. 我认为最重要的 6 类额外特性

下面这 6 类特性，才是我们真正应该拿来定义【控制系统状态机】的核心差异项。

---

### 4.1 模式化与层次化的控制组织

#### 这是什么

控制系统状态机往往不是扁平状态表，而是：

1. 上层管理工作模式、任务阶段、控制权限。
2. 下层管理局部动作和细粒度执行过程。
3. 子状态和父状态之间存在明确的进入、退出和边界语义。

典型例子包括：

1. 自动 / 手动 / 安全停机 / 故障恢复。
2. 起飞 / 航路 / 着陆。
3. 进路申请 / 锁闭 / 放行 / 释放。

#### 它解决什么痛点

控制系统在需求阶段最容易出现的问题之一，就是：

1. 模式切换逻辑散落在文档不同地方。
2. 局部动作和全局模式搅在一起。
3. LLM 或人工建模时容易把“阶段”与“子动作”混成同一层。

层次化和模式化能解决的核心痛点是：**把复杂控制逻辑拆成可管理、可解释、可验证的监督层和执行层。**

#### `pyfcstm` 能否支撑

可以，而且这是它当前最强的部分之一。

`pyfcstm` 已直接支持：

1. 层次状态机和复合状态。[`pyfcstm/README.md`](../../../pyfcstm/README.md)
2. 进入/退出子状态的初始转换和退出转换。[`docs/source/tutorials/dsl/index_zh.rst`](../../../pyfcstm/docs/source/tutorials/dsl/index_zh.rst)
3. 复合状态边界上的 `during before/after` 语义。[`docs/source/tutorials/simulation/index_zh.rst`](../../../pyfcstm/docs/source/tutorials/simulation/index_zh.rst)

所以这一块不是未来目标，而是我们现在就能做。

---

### 4.2 守卫化的联锁与许可控制

#### 这是什么

控制系统状态机中的很多转换，不是“某事件来就跳”，而是“某事件来且满足许可条件才跳”。  
这些许可条件通常不是业务校验，而是控制约束，例如：

1. 冲突进路不能同时保留。
2. 超载时不能起动。
3. 没有有效血压读数时不能保持自动控制。
4. 故障未消除时不能重新进入自动模式。

#### 它解决什么痛点

控制系统最大的问题之一，是大量安全约束、互锁条件、使能条件只是以自然语言散落出现：

1. 人工编码容易漏。
2. 需求转模型时容易少 guard。
3. 验证失败后很难定位是状态问题还是 guard 问题。

所以这类特性的作用，是把“**控制许可条件**”稳定地挂到迁移上。

#### `pyfcstm` 能否支撑

可以，且不需要深扩展。

`pyfcstm` 现有就支持：

1. transition guard。
2. 表达式系统。
3. 按定义顺序生效的转换优先级。

这意味着我们完全可以把很多 interlock / enable / inhibit 语义直接建模成 guard，而不需要再发明新的状态机构件。

对于博士课题而言，更关键的是：

1. 在数据抽取和生成阶段，把哪些自然语言条件识别成“控制 guard”。
2. 在验证阶段，把 guard 缺失、guard 错误作为缺陷类型单独处理。

---

### 4.3 异常、降级与恢复链

#### 这是什么

一般状态机当然也能画 `Error` 状态，但控制系统状态机更强调：

1. fault 如何进入。
2. 降级模式是什么。
3. 手动接管如何发生。
4. reset / restart / recalibration 的恢复链如何组织。

换句话说，异常路径不是边角料，而是主体。

#### 它解决什么痛点

实际控制系统里最不容易建对的往往不是 nominal path，而是：

1. 故障发生后系统去哪。
2. 需要人工参与还是自动恢复。
3. 何时允许重新返回正常模式。

如果这部分没有被显式建模，后续验证、修复、故障注入都做不扎实。

#### `pyfcstm` 能否支撑

可以，而且有两个现成抓手：

1. 普通 fault / recovery transition 可以直接建。
2. `! * -> ErrorHandler` 这类强制转换语法糖，非常适合表达“从多个运行状态统一跳向保护/异常处理态”。[`docs/source/tutorials/dsl/index_zh.rst`](../../../pyfcstm/docs/source/tutorials/dsl/index_zh.rst)

所以在现阶段，我们完全可以把“控制系统状态机”的异常语义主要建模为：

1. fault event。
2. fault guard。
3. 强制转换或统一恢复路径。
4. 进入/退出动作中的 reset / cleanup / acknowledge。

这一层已经够支撑很多工业控制样本，不必先引入复杂 fault meta-model。

---

### 4.4 分层事件与可控事件作用域

#### 这是什么

控制系统里的事件通常不是同质的。至少可以分成：

1. 操作员命令。
2. 传感器/检测事件。
3. 内部控制事件。
4. 跨层共享事件。
5. 全局故障/急停事件。

在一般状态机里，事件常常只是名字；但在控制系统里，事件归属、共享范围和传播范围非常重要。

#### 它解决什么痛点

控制系统建模常见问题是：

1. 同名事件在不同局部语境下含义不同。
2. 某事件应是局部按钮触发，结果被误做成全局事件。
3. 某故障应广播到上层 supervisor，结果只在局部状态里可见。

所以事件命名空间不是语法细节，而是控制语义清晰性的关键部分。

#### `pyfcstm` 能否支撑

可以，而且这是 `pyfcstm` 非常有辨识度的一张牌。

它已经支持三种事件作用域：

1. 本地事件 `::`
2. 链事件 `:`
3. 绝对事件 `/`

这使得我们可以不靠重扩展，就把控制系统里的事件分层关系组织得比较清楚。[`docs/source/tutorials/dsl/index_zh.rst`](../../../pyfcstm/docs/source/tutorials/dsl/index_zh.rst)

因此，从“控制系统状态机”的角度看，`pyfcstm` 的事件作用域机制本身就已经是一个很强的领域支撑点。

---

### 4.5 明确、可执行的生命周期语义

#### 这是什么

控制系统状态机不只是“图结构”，而是要能回答：

1. 进入状态时做什么。
2. 驻留时做什么。
3. 离开时做什么。
4. 父状态对所有子状态的通用动作何时执行。
5. 复合状态边界动作和叶状态动作谁先谁后。

#### 它解决什么痛点

很多主流状态机工具在工程上最大的问题不是“不能表示”，而是：

1. enter/during/exit 语义说不清。
2. 复合状态边界动作不清楚。
3. 实际执行顺序和设计人员想的不一致。
4. 仿真、验证、代码生成之间语义漂移。

控制系统比普通业务流程更怕这种漂移，因为动作次序本身就是控制逻辑的一部分。

#### `pyfcstm` 能否支撑

可以，而且这是它最适合拿来支撑博士课题的重要原因之一。

`pyfcstm` 已经显式支持：

1. `enter / during / exit`。
2. `>> during before/after` 切面式祖先动作。
3. 复合状态 `during before/after` 的边界语义。
4. 仿真 runtime 中的明确执行顺序。
5. 对 non-stoppable state 的 DFS 可达性验证。

这些能力意味着：  
我们不只是能“画控制系统状态机”，还能把它当作**有明确定义执行语义的控制状态机**来使用。[`docs/source/tutorials/simulation/index_zh.rst`](../../../pyfcstm/docs/source/tutorials/simulation/index_zh.rst) [`SIMULATE_DESIGN.md`](../../../pyfcstm/SIMULATE_DESIGN.md)

---

### 4.6 面向验证的建模友好性

#### 这是什么

控制系统状态机相对一般状态机的最后一个关键额外特性，不是新的建模元素，而是：

1. 结构必须适合仿真。
2. 结构必须适合形式化验证。
3. 缺陷必须能回溯到状态、guard、转换链等具体位置。

#### 它解决什么痛点

很多 UML/SysML 状态机在工程上最大的问题不是图画不出来，而是：

1. 模型可视化了，但验证不友好。
2. 语义模糊，无法稳定映射。
3. 发现问题后不知道是哪个元素错了。

控制系统开发真正缺的是“能服务验证闭环的状态机”，而不是“能展示行为的状态机”。

#### `pyfcstm` 能否支撑

可以支撑到一个很有价值的程度。

当前 `pyfcstm` 已有：

1. 仿真运行时。
2. 热启动与状态路径控制。
3. 对转换链可达性的验证语义。
4. 基于 Z3 的形式化验证设计雏形。

所以，若把我们的课题目标收敛为“**离散监督控制层的状态机生成-仿真-验证**”，那么 `pyfcstm` 已经足够构成一个研究内核，而不需要先去追求一整套大而全的新形式体系。[`pyfcstm/pyfcstm/verify/DESIGN.md`](../../../pyfcstm/pyfcstm/verify/DESIGN.md)

## 5. 基于 `pyfcstm` 这副牌，我们现在能做什么样的控制系统状态机

上面说的是“应该有哪些额外特性”。下面更关键的是：在不深扩 `pyfcstm` 的前提下，我们现实中能做什么。

我认为，基于现在的 `pyfcstm`，我们完全可以把目标控制系统状态机收敛为：

> **面向离散监督控制层的层次化控制状态机**

它尤其适合下面这些对象：

1. 电梯门控与运行模式。
2. 交通灯相位和优先放行。
3. 铁路联锁进路生命周期。
4. 输液泵/呼吸机的模式切换与保护逻辑。
5. PLC 设备模块的自动/手动/复位/安全停机。
6. 飞行任务 supervisor 的阶段切换与接管逻辑。

这些对象的共同点是：

1. 主要矛盾在模式、阶段、切换、互锁、恢复。
2. 连续控制律不是核心建模对象，或者只通过阈值和条件间接进入状态机。
3. 需要可解释执行语义。
4. 需要后续仿真和验证。

这和 `project_1/sources` 里当前命中的高价值样本高度一致。

## 6. 在不深扩 `pyfcstm` 的前提下，建议采用的建模口径

如果我们不打算一开始就大改 `pyfcstm`，那【控制系统状态机】应主要通过下面这些方式落地。

### 6.1 把“控制系统特性”落在建模规范上，而不是立刻落在新语法上

优先做法：

1. 用层次状态表达模式和阶段。
2. 用 guard 表达联锁和许可。
3. 用强制转换表达故障统一跳转。
4. 用事件作用域表达命令/局部/全局事件层次。
5. 用 enter/during/exit 表达初始化、驻留控制和退出清理。

也就是说，先把“控制系统状态机”做成**领域化使用方式**，而不是先做成“全新 DSL 语法集合”。

### 6.2 把时间先当作建模约定，而不是 DSL 第一类语法

时间肯定是控制系统痛点，但在现阶段我不建议为了时间直接深扩 `pyfcstm`。

更务实的做法是先把时间语义分成两类：

1. **可直接用变量 + guard + event 表达的**
   - timeout
   - counter-based dwell
   - periodic trigger
   - deadline flag
2. **需要后续验证层解释的**
   - 真实 clock constraint
   - invariant
   - 更接近 timed automata 的时钟语义

这样可以保证我们先把大多数控制样本吃下来，而不是在时间扩展上卡住整个研究。

### 6.3 把故障/安全语义先当作状态与转换角色解释

现阶段也不必把 `fault`, `safe`, `degraded` 做成 DSL 新关键字。  
可以先通过研究口径约定：

1. 哪些状态名属于 fault/degraded/safe/manual/auto。
2. 哪些转换属于 recovery/protection/interlock。
3. 哪些事件属于 command/sensor/fault/global emergency。

这足以支撑：

1. 数据标注。
2. LLM 生成提示。
3. 后续验证剖面构造。
4. 缺陷分类。

## 7. 哪些东西现在不要作为主目标

为了避免讨论失焦，我建议明确列出“现在先不追”的部分。

### 7.1 不把完整连续控制律并入状态机主体

例如：

1. PID/滑模/观测器方程。
2. 连续动力学状态空间模型。
3. 完整 hybrid automata 级别 plant dynamics。

这些东西当然重要，但不应成为 `project_1` 建模主线的中心对象。

### 7.2 不把 `pyfcstm` 立即扩成大而全的工业状态机标准

例如：

1. orthogonal regions。
2. history states。
3. 复杂端口系统。
4. 完整 fault meta-model。
5. 一整套时间自动机原生语法。

这些不是完全不做，而是不应成为当前定义【控制系统状态机】时的前置条件。

### 7.3 不把“控制系统状态机”定义得比 `pyfcstm` 当前能力宽太多

这是最重要的一点。

如果我们把课题对象定义成“完整工业控制系统的一切离散+连续+并发+历史+时钟+容错语义统一建模”，那么：

1. `pyfcstm` 会显得不够。
2. 讨论会自然滑向大幅扩 DSL。
3. `project_1` 当前数据建设也会被迫扩边界。

这不是你现在要的路线。

## 8. 一个更贴近当前研究阶段的定义建议

综合上面所有讨论，我建议在当前阶段把【控制系统状态机】定义成：

> 一类面向离散监督控制层的层次化状态机。  
> 它相较于一般状态机，额外强调模式组织、联锁守卫、异常恢复、分层事件、明确执行语义和验证友好性；  
> 在实现上，以 `pyfcstm` 为状态机内核，通过既有的层次状态、守卫、效果、生命周期动作、强制转换、事件作用域、仿真与验证机制来承载这些控制系统特性。

这个定义有三个好处：

1. 它真的能解释“我们和一般状态机差在哪”。
2. 它真的能解释“这些差异解决了控制系统哪些痛点”。
3. 它不要求我们先把 `pyfcstm` 做大改，和你手上的牌是对齐的。

## 9. 最终结论

如果把这份讨论稿压成一句最核心的话，我现在建议写成：

> 我们的【控制系统状态机】不是试图另造一种完全脱离现有状态机理论的新模型，而是在一般层次化状态机之上，突出模式控制、联锁约束、故障恢复、事件分层、明确执行语义和验证友好性的一类状态机；在工程实现上，它以 `pyfcstm` 为核心支撑，通过已有 DSL 与执行/验证机制承载控制系统最关键的离散监督逻辑。

这才是当前最稳、最能落地、也最符合你现有研究资源配置的说法。
