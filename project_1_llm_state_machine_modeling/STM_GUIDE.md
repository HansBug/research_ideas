# STM.md 撰写规范

本文档用于固定 `project_1_llm_state_machine_modeling/sources/*/STM.md` 的当前写法、内容要求和判定逻辑。凡是从给定文献中提取“带有状态机要素的控制系统设计/描述/需求等自然语言内容”，都应遵循本规范。

## 1. 目标与适用范围

`STM.md` 的目标不是复述整篇论文，也不是泛泛总结“论文用了状态机”。它的目标是：

1. 从论文原文中找出**真实控制系统**的设计描述、功能描述或需求描述。
2. 这些内容应当能够支撑后续构建**状态机自然语言需求/描述数据集**。
3. 每一条内容都必须能回溯到论文中的**具体页码、章节/图名、摘录原文**。

本规范适用于两种情况：

1. 论文中找到了可用的控制系统描述，此时需要写出一个或多个“条目”。
2. 论文中没有找到可用内容，此时也要留下“未收获”的检查记录。

## 2. 什么算目标对象，什么不算

### 2.1 算目标对象

优先提取以下对象：

1. 实际控制系统，如输液泵、起搏器、恒温器、自动变速器、刹车系统、飞机轮刹系统、分拣机、PLC 设备模块等。
2. 实际控制系统中的子系统、控制器、功能块、执行单元，只要它们确实承担控制职责。
3. 具有明确运行模式、阶段推进、条件切换、故障降级、恢复路径、时序约束的控制逻辑。
4. 用自然语言、状态图、自动机、TA/EFSM/ECC/Petri Net 等形式表达，但能够整理成控制系统自然语言描述的内容。

### 2.2 不算目标对象

以下内容通常不应提取为条目：

1. 开发流程、需求分析流程、验证流程、建模流程、代码生成流程。
2. 工具自身的模式、界面状态、工作流状态。
3. 纯方法论、纯架构分层、纯元模型说明，但没有实际控制对象行为描述。
4. “论文/实验/教学流程”的阶段，不是被控系统或控制器本身的运行逻辑。
5. 只能看出“有状态”但状态属于文档处理、需求管理、工具使用，而不是控制系统。

### 2.3 判定口径

判断“算不算我们关注的具有状态机属性的控制系统”时，至少同时检查以下三点：

1. **对象真实性**：对象必须是控制系统、控制器或控制子系统，而不是流程或工具。
2. **行为可组织性**：原文中要能看出模式切换、条件驱动、阶段推进、故障处理、时序约束或类似状态机属性。
3. **证据可追溯**：最终写入 `STM.md` 的每句话都必须能在原文里找到依据。

## 3. 提取前的阅读与证据原则

### 3.1 阅读顺序

推荐顺序如下：

1. 先看同目录下的 `desc.md`，快速把握论文主题。
2. 再精读 `paper_content.txt`，以此作为 `STM.md` 的主要证据来源。
3. 如 `paper_content.txt` 提取质量差，优先用 `tools/pdf_extractor.py` 重新提取；必要时改用 OCR。
4. 如文本仍明显异常，应回到 PDF 原文核对，不得在证据不足时硬写。

### 3.2 证据原则

1. `STM.md` 中所有判断、摘录、整理后的自然语言内容都必须以原文为依据。
2. 可以做**合理转接**，把原文中分散的句子整理成连续描述，但不能加入找不到原文依据的新信息。
3. 如果原文有 OCR 噪声，摘录时允许保留噪声；不要为了“更顺”而擅自改写引文内容。
4. 如果控制逻辑只出现在图题、图注、变量表、公式说明中，这些也可以作为证据，但仍需写清出处。

```markdown
## 条目 1: [英文短标题]
- 控制对象：[中文说明]

### 0. 条目识别与判定

- 一句话说明：[必须讲清楚这是“什么领域里的什么系统/子系统/控制器”]
- 判断：[必须明确写“算”或“不算”，并说明是否属于我们关注的具有状态机属性的控制系统]

1. 句子 1：[完整复写第 2 节中的第 1 句]
```


## 5. 各部分的具体写法

### 5.1 `## 盘点结论`

这是文件级的总判断，不是单条目的判断。

字段含义如下：

1. `评级`
   - `🟢 直接可用`：控制对象明确，原文证据充分，能直接整理成高质量自然语言描述。
   - `🟡 可整理`：控制对象大体明确，但证据分散、图中文字较多、状态名不完整或摘录质量一般，仍可整理。
   - `⚪ 未收获`：没有找到符合要求的控制系统自然语言设计/描述/需求内容。
2. `是否计入 SOURCES.md 盘点`
   - 有实质条目时通常写“是”。
   - 未收获时写“否”。
3. `提取条目数`
   - 等于本文件中 `## 条目 n` 的数量。
4. `简要判断`
   - 1 句话概括整篇论文的提取价值或失败原因。

### 5.2 `### 0. 条目识别与判定`

这是每个条目必须具备的入口说明。

要求如下：

1. `一句话说明` 必须写清楚：
   - 所属领域；
   - 具体系统/子系统/控制器是什么；
   - 该对象在系统中负责什么控制任务。
2. `判断` 必须明确：
   - 这个对象到底算不算我们关注的控制系统；
   - 为什么算，或为什么只是“构件级样本”；
   - 为什么不算的情况，一般应放在“未收获”文件中，而不是正例条目里。

推荐写法：

1. “这是汽车底盘控制领域的自动变速器控制器，用于……”
2. “这是医疗设备领域的双腔起搏器控制系统，用于……”
3. “算。对象是实际控制系统，原文给出了……”
4. “算，但属于控制软件构件级样本……”

### 5.3 `### 1. 原文摘录`

这是最关键的证据区。

要求如下：

1. 每个摘录块必须有编号，如 `摘录 A / B / C`。
2. 每个摘录块必须写清出处：
   - 第几页；
   - 什么章节、图、表或段落；
   - `paper_content.txt` 中的行号范围。
3. 摘录内容使用 Markdown 引用块，即 `> ...`。
4. 摘录应尽量**完整、长一些、多一些**，宁可稍长，不要只摘一个关键词。
5. 摘录时优先保留原文上下文，以便后面的自然语言整理和逐句溯源有足够依据。
6. 若不同句子的依据分散在多个位置，应保留多个摘录块，不要强行拼成一段伪引文。

### 5.4 `### 2. 基于原文整理后的自然语言描述`

这是最终要服务数据集构建的核心内容。

要求如下：

1. **默认用英文撰写**，因为目标通常是尽量保留原始论文的表达方式。
2. 必须基于第 1 节摘录内容整理，不得越界扩写。
3. 在允许的情况下，尽量直接使用原文或原描述的措辞。
4. 允许的改写仅限于：
   - 增加必要连接词；
   - 把分散在不同段落中的原文句子串起来；
   - 对明显重复的表达做轻微去重。
5. 不允许：
   - 引入原文没有的功能、变量、条件、阈值、模式名称；
   - 加入自己的解释性假设；
   - 把图中没出现的状态数量、迁移数量硬写出来。
6. 行文上应写成“系统设计/功能/需求/描述”式自然语言，不要过于直白地写成：
   - “该状态机有 4 个状态”
   - “发生事件 e1 后从 s1 转到 s2”
7. 但如果论文原文本身就以这种方式表述，而且这是必要事实，则可以谨慎保留。

推荐标准：

1. 读起来像一段系统设计说明或功能需求说明。
2. 句子之间连续自然。
3. 每一句都能在前面的摘录里找到依据。

### 5.5 `### 3. 逐句溯源`

这一节用于保证最终描述是“句句可追溯”的。

要求如下：

1. 将第 2 节中的每一句话**完整复写**出来。
2. 对每一句标注其对应的摘录编号，如 `A`、`A, B`、`B, C`。
3. 这里的对应关系必须能回到第 1 节的原文摘录块。
4. 如果一句话依赖多个摘录，必须全部列出。
5. 如果某一句无法找到对应摘录，说明该句不应留在第 2 节。

## 6. 内容组织逻辑

实际提取时，建议按下面逻辑走：

1. 先确定论文里出现的候选对象是不是控制系统。
2. 再判断这个对象是否具有可整理的状态机属性。
3. 再收集原文证据，不要先凭印象写结论。
4. 最后基于原文写连续自然语言描述，并逐句回溯。

换言之，逻辑顺序是：

`对象识别 -> 目标判定 -> 原文摘录 -> 整理描述 -> 逐句溯源`

而不是：

`先脑补一个控制逻辑 -> 再去找相似原文`

## 7. 什么时候应判定为“未收获”

出现以下情况时，应直接写成 `⚪ 未收获`：

1. 找到的“状态”属于开发流程、需求生命周期、工具模式、工作流步骤，而不是控制系统。
2. 论文只有架构层面的模块关系，没有行为、模式、切换、时序或故障处理描述。
3. 文中虽然提到 automata / state machine，但对象只是形式化框架说明，没有实际控制对象的设计内容。
4. 只有图，没有足够正文或图注支撑，无法形成可追溯自然语言描述。
5. `paper_content.txt` 质量太差，而重新提取或 OCR 后仍不足以支撑可靠摘录。

## 8. 与 `SOURCES.md` 的关系

当任务要求汇总时，`STM.md` 应作为 `SOURCES.md` 盘点的直接依据。

通常需要从 `STM.md` 中汇总的信息包括：

1. 该论文是否有收获；
2. 评级；
3. 提取条目数；
4. 控制对象类型；
5. 是否属于直接可用于状态机自然语言数据集的样本。

## 9. 当前仓库采用的风格总结

当前 `STM.md` 的写法固定为：

1. 标题、盘点结论、判定说明、出处说明用中文。
2. 条目标题通常用英文短语，便于快速概括控制逻辑。
3. `控制对象` 用中文写，强调对象是什么。
4. 原文摘录尽量保留英文原文。
5. 整理后的自然语言描述默认使用英文。
6. 逐句溯源用中文框架配英文句子。

后续新增或重写 `STM.md` 时，应保持这一风格一致。
