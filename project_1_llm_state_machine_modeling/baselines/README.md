# `baselines/` 论文集 README

## 1. 论文集定位

`project_1_llm_state_machine_modeling/baselines/` 是 `project_1` 下专门维护“LLM 生成/精化/扩展状态机模型 baseline”的论文集工作区。

它服务于博士研究中“基于 LLM 的控制系统状态机结构化建模方法”的对比基线建设，不负责沉淀控制系统原始样本，也不负责收录一般形式化方法背景文献。这里的核心目标，是把后续需要反复比较的“状态机生成/精化/修复”方法论文固定为一组可追溯、可对照、可持续扩展的 baseline 目录。

## 2. 设立宗旨与期望收获

单独建立本论文集，主要为了沉淀以下几类内容：

1. 与 `project_1` 直接可比的 LLM 状态机建模论文原文、完整 BibTeX、可检索文本与单篇分析。
2. 面向横向对比的统一字段：输入工件、输出工件、LLM、提示/反馈机制、验证方式、数据集来源、开放性、局限性。
3. 对后续实验设计、baseline 选择、研究空白论证真正有用的汇总观察，而不是零散检索笔记。

本论文集希望最终回答三类问题：

1. 当前有哪些工作真正做了“需求/图样/已有模型 -> 状态机模型”的 LLM 自动化建模？
2. 这些工作分别用了什么输入、输出、提示工程、反馈闭环、验证方式和 benchmark？
3. 它们在哪些方面仍不足以覆盖控制系统状态机建模的核心需求，例如时间约束、安全约束、层次并发语义和形式化验证闭环？

## 3. 收录范围

本论文集优先收录以下论文：

1. 使用 LLM 直接生成状态机、状态图、Statechart、SysML 状态机等状态机族模型的论文。
2. 使用 LLM 对已有状态机/状态图进行精化、扩展、修复、约束注入、测试生成或多模态识别，并且结果仍可作为 `project_1` 的对比基线。
3. 少量与状态机任务强邻近的论文，例如状态图识别到可机读状态机、有限状态推理 benchmark；这类论文只能作为弱相关补充，不是本论文集的主线。

原则上不应作为本论文集重点收录的论文：

1. `sources/` 那类真实控制系统样本文献，它们服务于数据集与案例来源，而不是 baseline 比较。
2. 纯综述、纯方法背景、纯工具介绍、纯形式化理论文献。
3. 虽然出现 `FSM` 或 `state machine`，但实际任务是 LLM 控制流程、对话流程、智能体编排，而不是软件/系统建模工件生成。
4. 输出不是状态机族模型的论文，例如类图、顺序图、目标模型、一般领域模型；除非它们能补充某个明确缺口，否则不应继续扩张收录。
4. 不提供足够原文证据，无法稳定整理出单篇 `DESC.md` 的论文。

## 4. 纳入与排除判定标准

后续判断一篇论文是否进入 `baselines/` 时，至少从以下维度执行：

1. 研究对象
   - 纳入：LLM 建模方法本身，且建模工件可与 `project_1` 形成横向比较。
   - 排除：控制系统案例论文、纯标准解读、纯工具综述。
2. 任务类型
   - 纳入：围绕状态机模型本体的生成、精化、扩展、修复、识别、转换、测试生成。
   - 降优先级：只做一般性代码生成，但仍显式围绕 FSM/状态图推理。
   - 排除：输出不是状态机族模型，或 FSM 仅作为提示范式/内部控制结构。
3. 证据形态
   - 纳入：原文能明确提取输入、输出、方法流程、实验设置、主要结果和局限性。
   - 排除：只有摘要级介绍，无法支持高质量 `DESC.md`。
4. 可提取性
   - 纳入：PDF 可获取，`paper_content.txt` 质量可用。
   - 排除：无法获得可用 PDF，或文本提取后仍不足以支撑可靠分析。
5. 与本研究相关性
   - 纳入：能直接用于状态机 baseline 比较、研究空白定位、评测设计或方法借鉴。
   - 排除：虽然与 LLM 或建模相关，但无法为 `project_1` 的状态机建模任务提供可执行参照。

## 5. 本论文集下文件说明

本论文集默认包含以下核心文件：

1. [README.md](./README.md)
   - 入口说明文件。
   - 负责解释本论文集存在的原因、收录边界和使用顺序。
2. [GUIDE.md](./GUIDE.md)
   - AI 工作操作规范。
   - 负责规定“只围绕状态机模型”的检索、筛选、目录维护、`DESC.md` 生产与 [SUMMARY.md](./SUMMARY.md) 回填流程。
3. [SUMMARY.md](./SUMMARY.md)
   - 论文集总账。
   - 记录当前收录数、分类盘点、关键词簇、论文清单、待补充候选和更新日志。
4. [DESC_GUIDE.md](./DESC_GUIDE.md)
   - 单篇 `DESC.md` 的专项规范。
   - 规定 `DESC.md` 必备章节、比较维度、证据要求与“未获取信息”的写法。

AI 推荐阅读顺序如下：

1. [README.md](./README.md)
2. [GUIDE.md](./GUIDE.md)
3. [SUMMARY.md](./SUMMARY.md)
4. [DESC_GUIDE.md](./DESC_GUIDE.md)（当任务涉及单篇 `DESC.md`）
5. 目标论文目录下的 `bibtex.bib`
6. 目标论文目录下的 `paper_content.txt`
7. 必要时回到 `paper.pdf`

## 6. 单论文路径约束

本论文集下每个单论文目录默认至少应包含：

1. `paper.pdf`
2. `paper_content.txt`
3. `bibtex.bib`
4. `DESC.md`

其中：

1. `paper_content.txt` 必须优先使用 `tools/pdf_extractor.py` 生成。
2. `DESC.md` 是本论文集的核心派生文件，必须遵循 [DESC_GUIDE.md](./DESC_GUIDE.md)。
3. `DESC.md` 必须显式写清代码/仓库获取方式和数据集/benchmark 获取方式；若原文未提供，需要如实写明。

## 7. AI 工作入口提示

进入本论文集时，默认按以下方式工作：

1. 先读 [README.md](./README.md)，确认收录边界，避免把 `sources/` 类样本论文误收进来。
   并且明确：非状态机输出论文默认不是重点对象。
2. 再读 [GUIDE.md](./GUIDE.md)，确认这一轮的检索、筛选和回填流程。
3. 再读 [SUMMARY.md](./SUMMARY.md)，掌握当前已收录范围、比较口径与待补充候选。
4. 若需要创建或重写单篇分析，必须再读 [DESC_GUIDE.md](./DESC_GUIDE.md)。
5. 进入单论文目录后，严格按 `bibtex.bib -> paper_content.txt -> paper.pdf（必要时） -> DESC.md` 的顺序处理。
6. 完成后必须回写 [SUMMARY.md](./SUMMARY.md)，不能只在论文目录里留下未入账条目。

## 8. 后续 AI 应优先做什么、避免做什么

优先做的事：

1. 补齐当前高价值 baseline 的 `DESC.md` 和汇总字段一致性。
2. 优先收录那些能补强“控制系统状态机建模”比较维度的工作，例如带约束精化、反馈修复、多模态识别、验证闭环。
3. 用 `SUMMARY.md` 固定高命中关键词、类别边界和待补充候选，减少重复检索成本。

应避免的事：

1. 把所有“LLM + FSM”论文都机械收进来，导致 collection 漫无边界。
2. 把顺序图、类图、目标模型之类非状态机输出论文误当成直接 baseline。
2. 只看标题或摘要就断言某论文是强 baseline。
3. 新增论文目录而不更新 [SUMMARY.md](./SUMMARY.md)。
4. 只写泛泛综述，不回到原文提取可比较的输入/输出/方法/实验细节。
