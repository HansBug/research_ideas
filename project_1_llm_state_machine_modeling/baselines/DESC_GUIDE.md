# `baselines/` 的 `DESC.md` 编写规范

本文档用于规范 `project_1_llm_state_machine_modeling/baselines/*/DESC.md` 的写法。它服务于 `project_1` 的 baseline 建设，目标不是写泛泛的读书笔记，而是产出一份能直接用于横向比较、研究空白论证、实验设计和论文写作的单篇分析。

当前仓库中的 `DESC.md` 已经形成了一套比较稳定的风格。后续自动生成的新 `DESC.md`，默认必须向这套格式收敛，而不是自由发挥。历史文件若存在个别章节名差异，可暂时兼容，但新文件应优先遵循本规范。

## 1. 使用前提与阅读顺序

开始编写单篇 `DESC.md` 前，必须先完成以下阅读顺序：

1. [README.md](./README.md)
2. [GUIDE.md](./GUIDE.md)
3. [SUMMARY.md](./SUMMARY.md)
4. 当前论文目录下的 `bibtex.bib`
5. 当前论文目录下的 `paper_content.txt`
6. 必要时核对 `paper.pdf`

如果该论文已在 `project_1` 根目录的历史工作文档中出现，例如 `BASELINE.md`、`BASELINE_TO_EXPLORE.md`、`DIR_1_llm_direct_state_machine_generation.md`、`LITERATURE_SEARCH_RESULTS.md`，可以把这些文档当作线索来源，但不能替代原文阅读。

## 2. `DESC.md` 的核心目标

每篇 `DESC.md` 至少需要回答以下问题：

1. 这篇论文做的是什么任务，和 `project_1` 的目标关系有多直接。
2. 输入到底是什么，输出到底是什么，输出属于什么模型或工件类型。
3. 论文用了什么 LLM、提示工程、反馈机制、验证手段和数据集。
4. 这篇论文应该被评为 `🟢/🟡/🟠/⚪` 中的哪一类 baseline 关系。
5. 它对 `project_1` 的真正借鉴点和真实局限性分别是什么。

这里额外固定一条 collection 级硬约束：

1. 若论文输出不是状态机、Statechart、SysML 状态机或其他高度等价的状态机族模型，则 **不得评为 `🟢`**。
2. 顺序图、类图、goal model、一般 domain model 等论文，默认只能评为 `🟠`，除非它们实际围绕已有状态机做精化/修复，此时才可考虑 `🟡`。

## 3. 标题与总格式

### 3.1 文件标题

文件开头默认使用一级标题：

```markdown
# 中文标题 / English Title
```

规则如下：

1. 优先使用“中文标题 / 英文标题”的双语格式。
2. 若论文没有官方中文标题，可以给出忠实、克制的中文翻译。
3. 不要只写英文标题，除非中文翻译确实会引入明显歧义。

### 3.2 一级章节顺序

除非论文材料极端特殊，默认一级章节顺序应为：

1. `## 基本信息`
2. `## 简报`
3. `## 研究问题与动机`
4. `## 核心方法`
5. `## 实验与评估`
6. `## 与本研究的关系`
7. `## 重要的相关工作`
8. `## 文献分类总结`

允许按论文实际情况补充附加章节，例如：

1. `## 关键引用`
2. `## 实验细节补充`
3. `## 总结`

但这些附加章节只能放在主干章节之后，不能替代主干章节。

## 4. `## 基本信息` 的固定写法

该章节默认使用短条目列表，至少包含以下字段，顺序尽量固定：

1. `标题`
2. `中文标题`
3. `作者`
4. `单位`
5. `发表`
6. `DOI`
7. `链接`

紧接着必须显式写出两项获取信息：

1. `代码/仓库获取方式`
2. `数据集获取方式`

推荐写法如下：

```markdown
## 基本信息

- **标题**：
- **中文标题**：
- **作者**：
- **单位**：
- **发表**：
- **DOI**：
- **链接**：

**代码/仓库获取方式**：
- ...

**数据集获取方式**：
- ...
```

规则如下：

1. 若原文未提供公开代码或仓库链接，必须明确写“原文未提供公开代码/仓库获取链接”。
2. 若原文未提供公开数据集/benchmark 下载链接，必须明确写“原文未提供公开数据集获取链接”。
3. 若只能通过论文正文、附录、GitHub 工件、Zenodo、Google Drive 或联系作者获取，也要写清楚。
4. 这里强调的是“如何获取”，不是“如何构建”。

## 5. `## 简报` 的固定写法

简报是高密度概览区，必须回答以下五个问题，并保持顺序一致：

1. 解决的问题
2. 研究动机
3. 方法创新
4. 实验设计
5. 结论与不足

其中“解决的问题”必须满足：

1. 明确写出 `输入是什么`
2. 明确写出 `方法是什么`
3. 明确写出 `输出是什么`
4. 文字后必须附一个字符示意图，展示 `输入层 -> 方法层 -> 输出层`

在该章节中，推荐使用以下固定小结构：

1. 一段总述
2. `- 输入：`
3. `- 方法：`
4. `- 输出：`
5. 字符示意图
6. 一句实验结果总结

## 6. `## 研究问题与动机` 的固定写法

该章节默认至少包含以下二级标题：

1. `### 问题背景`
2. `### 核心问题`
3. `### 研究动机`

若材料足够，建议再补：

1. `### 研究意义`
2. `### 现有方法的局限性`
3. `### 研究目标`

这一章的核心任务不是复述摘要，而是讲清：

1. 论文面对的到底是什么建模或推理任务。
2. 为什么现有方法不够。
3. 它和 `project_1` 的目标在任务定义上是直接可比、部分可比还是只属邻近任务。

## 7. `## 核心方法` 的固定写法

该章节默认至少包含：

1. `### 方法概述`
2. 若干个方法子模块小节

常见可用二级/三级标题包括：

1. `### 关键技术`
2. `### 系统架构`
3. `### 形式化定义`
4. `### 技术实现细节`
5. `### 数据集构建`
6. `### 方法优势`

必须写清以下内容：

1. 总体 pipeline
2. 输入在方法内部如何被处理
3. 中间表示是什么
4. 输出格式是什么
5. 是否使用 few-shot / CoT / RAG / 自反思 / 自动反馈循环 / 约束注入 / 模型检查 / 仿真

如果论文包含多个阶段，例如 `Phase-I / Phase-II`、`生成 / 修复`、`识别 / 生成`，必须按阶段展开，不要混在一起。

## 8. `## 实验与评估` 的固定写法

该章节默认至少包含以下信息，标题可以按论文内容微调，但语义不能缺：

1. 数据集 / benchmark / 案例系统
2. 评估指标
3. 实验设置 / 研究问题
4. 主要实验结果
5. 方法优势
6. 方法局限性

推荐的二级标题集合：

1. `### 数据集`
2. `### 评估指标`
3. `### 实验设置`
4. `### 主要实验结果`
5. `### 方法优势`
6. `### 方法的局限性`

可根据论文内容再拆为：

1. `#### 实验1 / 实验2 / 消融实验`
2. `#### 案例一 / 案例二`
3. `#### 错误类型分析`
4. `#### 威胁有效性分析`

这一章必须尽量保留关键数字，尤其是：

1. 规模
2. 准确率 / F1 / 成功率 / Pass@K / 编辑距离
3. 对比提升幅度
4. 失败模式

## 9. `## 与本研究的关系` 的固定写法

该章节默认至少包含以下二级标题：

1. `### 相关性分析`
2. `### 可借鉴之处`
3. `### 存在的不足与改进空间`
4. `### 对本研究的启发`

若论文与 `project_1` 的差异很大，建议再补：

1. `### 研究定位与差异化`

这里必须显式给出 `BASELINE评估` 结论。建议在 `### 相关性分析` 中明确写：

1. 当前论文的 `BASELINE评估` emoji
2. 为什么是这一类
3. 它和 `project_1` 在输入、输出和任务目标上的主要差异

若论文输出不是状态机族模型，这里必须显式写出：

1. “为什么不能算直接 baseline”
2. “为什么仍然保留”

## 10. `## 重要的相关工作` 的固定写法

这是 `DESC.md` 的核心章节之一，必须基于原文的相关工作与参考文献来写，不能编造。

默认按以下五类组织：

1. `### 1. 重要的前身类工作`
2. `### 2. 直接参与实验的baseline`
3. `### 3. 提供了重要论证的工作`
4. `### 4. 在技术上提供了支持的工作`
5. `### 5. 其他重要工作`

若原文存在特殊情况，可以增加一类，例如：

1. `### 5. 数据集来源文献`

但新增类别必须确实来源于论文内容，而不是为了凑结构。

对每篇重要文献，建议尽量包含：

1. 标题
2. 作者/年份
3. 主要内容
4. 论文中的引用位置或引用语义
5. 与本论文的关系
6. 它提供了什么支撑

## 11. `## 文献分类总结` 的固定写法

每篇 `DESC.md` 收尾前默认应有 `## 文献分类总结`，用于对“重要的相关工作”做收束。

本节应简要总结：

1. 当前论文位于什么研究链条上
2. 它依赖了哪些前身、技术、论证和数据来源
3. 它在 baseline 体系中的定位是什么

## 12. 必须显式抽取的比较字段

为便于后续汇总，`DESC.md` 中必须尽量显式回答以下字段：

1. `输入`
2. `输入形态`
3. `输出`
4. `输出模型类型`
5. `使用的LLM`
6. `主要方法`
7. `是否使用 few-shot / CoT / RAG / 自动反馈 / 修复闭环`
8. `是否包含形式化验证、模型检查、仿真或约束注入`
9. `是否面向控制系统或安全关键系统`
10. `数据集/benchmark 是否公开`
11. `代码/仓库是否公开`
12. `BASELINE评估`

其中对 `输入/输出` 的要求特别严格：

1. 不得只写“文本”“模型”“代码”。
2. 要尽量写成“自然语言需求”“自然语言 + I/O 接口规范”“初始 FSM + STPA 约束”“状态图图像”等。
3. 输出要尽量写成“SysML 状态机”“Mermaid Statechart”“Umple 状态机”“SystemVerilog FSM 代码”“扩展状态图 + 测试用例”等。

## 13. 允许的可选章节与边界

在不破坏主结构的前提下，可按论文实际情况增加以下章节：

1. `## 关键引用`
2. `## 实验细节补充`
3. `## 总结`

但以下做法不推荐：

1. 用自创章节替代 `## 与本研究的关系`
2. 省略 `## 简报`
3. 省略 `## 重要的相关工作`
4. 把所有实验内容散落在方法章节里

## 14. 证据与行文要求

编写 `DESC.md` 时，必须遵守以下要求：

1. 结论必须基于 `paper_content.txt` 或 PDF 原文。
2. 若提取文本有歧义、图表缺失或公式断裂，应回到 PDF 核对。
3. 对关键数字、评价、局限性、相关工作关系，尽量给出可追溯证据。
4. 可做综合分析，但不能凭常识补写原文没有的事实。
5. 若原文没有给出某项信息，明确写“原文未明确说明”。

## 15. 与 collection 级文档的分工

以下内容应放在 `DESC.md`：

1. 单篇论文的完整分析。
2. 单篇论文的输入/输出/方法/数据集细节。
3. 单篇论文的 baseline 关系判断。
4. 单篇论文的相关工作脉络。

以下内容应主要放在 [SUMMARY.md](./SUMMARY.md)：

1. 跨论文统计。
2. `BASELINE评估` 分布。
3. 数据集与 benchmark 总表。
4. 待补充候选与整体更新日志。

## 16. 冲突处理

如果 [GUIDE.md](./GUIDE.md) 与本文件发生冲突：

1. collection 的收录边界、目录回填与汇总口径，以 [GUIDE.md](./GUIDE.md) 为准。
2. 单篇 `DESC.md` 的章节结构、比较字段和写法，以本文件为准。
