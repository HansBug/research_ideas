## 简报

这篇论文把 LLM 当作教学和新手建模助手，观察其在 use case、class diagram、sequence diagram 三类 UML 模型中的辅助效果。它更像经验研究，而非新方法论文。

- **输入**：给定 case study、学生 prompts 与 LLM 对话
- **方法**：45 名学生在课程项目中使用 ChatGPT 完成 UML 建模
- **输出**：用例图、类图、顺序图及对话记录分析

```text
需求案例 -> 学生与 ChatGPT 交互 -> UML 模型初稿/修订 -> 教学实验分析
```

它说明 LLM 已经能在 UML 建模中扮演“助教”角色，但离自动状态机建模还有距离。

## 研究问题与动机

### 问题背景

需求建模和 UML 建模对新手而言耗时且容易出错，而 LLM 在自然语言交互上的优势使其很适合作为辅助工具。

### 核心问题

1. LLM 能在多大程度上帮助新手完成 UML 建模。
2. 哪些模型类型更容易得到 LLM 帮助。
3. 输出格式、prompt 与对话方式会如何影响结果。

### 研究动机

作者希望通过真实课程实验而非小样本演示，观察 LLM 在 UML 建模中的实际使用形态。

## 核心方法

### 方法概述

研究要求 45 名软件工程本科生在课程项目中使用 LLM 完成三类 UML 模型，并提交模型与对话记录。研究者随后从 correctness、输出格式和交互模式等角度分析结果。

### 关键技术

1. **真实教学场景**。
2. **三类 UML 模型同时观察**。
3. **对话记录分析**：不仅看成品，也看交互过程。

## 实验与评估

### 数据集

- 45 份课程项目报告。
- 三类 UML 模型：use case、class diagram、sequence diagram。

### 主要实验结果

1. LLM 对 UML 建模有明显辅助作用。
2. 其优势主要体现在模型初稿和修改建议层面。
3. 输出格式和 prompt 组织方式会显著影响最终质量。

### 方法的局限性

1. 研究对象是新手学习场景。
2. 主要使用 ChatGPT，缺少多模型严谨横评。
3. 不包含状态机模型。

## 与本研究的关系

### 相关性分析

- **BASELINE评估**：🟠
- **原因**：论文是教学/经验研究，且输出不含状态机，只能作为“LLM 辅助 UML 建模能力”旁证。

### 可借鉴之处

1. 输出格式对结果的影响值得在状态机生成中单独控制。
2. 对话记录分析适合挖掘用户与 LLM 的协同模式。
3. 说明 LLM 作为“建模助手”比“完全自动建模器”更贴近现实使用。

### 存在的不足与改进空间

1. 没有控制系统场景。
2. 没有状态机。
3. 没有形式化验证或闭环修复。

### 对本研究的启发

可以把这篇论文用作“人机协同建模”一侧的证据，而不是自动状态机 baseline。
