Abstract —Large Language Models (LLMs) are increasingly
being used in software development and in applications like
code generation. While LLMs can provide significant value in
the form of time savings in common programming languages
like Python, their usability in generating automation software
has yet to be studied extensively. In the context of generating
control software in the form of IEC 61131-3 compliant code,
initial studies suggest LLMs provide a promising avenue for
increasing control engineer productivity. However, similar code
generation for IEC 61499-based control applications is still
scarce. While tools are being developed for this purpose, their
capabilities are not yet fully understood, and they often require
significant human input to generate the intended outcomes. This
paper explores LLM-based code generation for IEC 61499-based
applications through iterative prompting. The prompts for the
experiments are derived from requirements generated by System-
Theoretic Process Analysis (STPA), which provides a systematic
approach to creating prompts that also connect to the larger
systems engineering workflow. The results indicate that while
the approach may be successful in some instances, more work is
required to mitigate the issues arising from its application.
Index Terms —STPA, IEC61499, iterative prompting, LLM,
finite-state machine
I. I NTRODUCTION
Artificial Intelligence (AI) and Large Language Mod-
els (LLMs) have become increasingly popular over the past
few years, with common applications ranging from generating
images to analyzing data, and programming software. LLMs
are capable of generating code in programming languages
such as Python, and even Structured Text for IEC 61131-3
based control applications [1] [2] [3]. Research on the use
of LLMs in code generation use cases has also indicated
promising results [4], with some studies reporting significant
time savings [5]. While the use of LLMs in the generation of
code is increasing, their use in control automation software,
and specifically IEC 61499 is still rather limited. However,
steps are being taken towards this direction, as tools are being
developed to utilize LLMs in the context of generating IEC
61499 based automation software [6].
While LLM-based code generation in IEC 61499 has be-
come possible and increasingly sophisticated, it still requires
significant human input during the generation process to
generate functional control software. In fbAssistant, a proof-
of-concept tool currently being developed for this exact ap-plication [7], this input takes the form of prompts, through
which the LLM is guided on the changes it must implement
to the Finite State Machine (FSM) being created. Each prompt
may outline changes or conditions that the FSM must satisfy.
Currently, prompts may introduce unexpected changes in the
generated code, and the types of prompts that produce the best
results and the most suitable prompting strategies have yet to
be identified.
One approach to improving the results generated by LLMs
is iterative prompting, with certain iterative prompting strate-
gies demonstrating an increase in truthfulness over the course
of 10 iterations [8]. Could similar improvements in results
be observed when applying iterative prompting strategies to
generating control software with LLMs? Iterating the software
with the same prompt or evolving the software with the same
prompt could potentially lead to the control software iterations
converging to a solution that aligns better with the intended
behavior of the control software. Yet, little research has been
conducted on this aspect of code generation. If such prompting
strategies result in the iterations improving toward a better
solution, the approach could be used to generate functional
control software that aligns with the intended behavior with
little to no human intervention.
However, even if a convergence with certain types of
prompts can be observed, a systematic approach to generating
such prompts needs to be defined for the potential convergence
to be utilized effectively. In this context, System-Theoretic
Process Analysis (STPA) could provide a good source of
well-defined prompts, and enables the integration of control
software generation to systems engineering processes.
This paper explores an iterative prompting method for
generating control software in IEC61499-based control ap-
plications using STPA controller constraints and an LLM-
based FSM generation tool, fbAssistant. Of specific interest
are how different types of prompts impact the generated code,
as well as the advantages and disadvantages of evolving FSMs
with repeated prompts. The rest of the paper is structured as
follows. In Section II, the relevant literature is introduced.
Section III describes how the experiment was undertaken. In
Section IV-B, the results of the experiment are presented and
discussed. Finally, in Section V a summary of the findings is
presented, and future work is discussed.

II. P RELIMINARIES
A. STPA
System-Theoretic Process Analysis (STPA) is a hazard
analysis method that provides an attractive starting point for
the design and development of new systems [9], as it is
able to identify hazards arising from design flaws and the
interaction of the components within a system. [10] Central
to the STPA method are Control Structures, which are dia-
grams depicting the relations between controller and controlled
processes through control actions and feedback. In a Control
Structure, the controllers within a system assert control over
controllers and controlled processes lower in the hierarchy
through Control Actions (CA) defined in the Control Structure,
while system elements lower in the hierarchy may provide
feedback to the elements higher in the hierarchy.
The STPA method consists of 4 steps: Determining the
purpose of the analysis, Modeling the Control Structure,
Identifying Unsafe Control Actions (UCAs), and Identifying
Loss Scenarios. For the purpose of the experiment presented
in this work, only the first 3 steps are required.
In Step 1 of the analysis, Losses, Hazards and System
Constraints are defined, as well as the system boundary. Losses
can be summarized as outcomes that are to be avoided, such
as the loss of life, damage to equipment or financial loss.
Hazards are states of the system that may lead to a Loss,
such as an airplane flying too close to terrain, or a nuclear
reactor exceeding a temperature limit. System-Constraints
are essentially statements that negate the Hazards, e.g. ”The
reactor core must not exceed a temperature of 850 degrees
Celsius”.
Step 2 of the analysis involves developing a Control Struc-
ture diagram. This is often an iterative process, and may be
returned to later after identifying flaws or issues in later steps
that need to be addressed in design. The Control Structure
can be modeled at various levels of abstraction, from simple
diagrams with just a few blocks often being used to depict
systems that could be modeled at a much more intricate level.
In this work, the Control Structure is modeled at a level of
abstraction that distinguishes between different actuators and
sensors of the pick-and-place machine. This Control Structure
is presented in Fig. 1.
Step 3 of the analysis is used to generate UCAs and
controller constraints. The former outline how and in which
context a Control Action may become unsafe. The ”how” is
categorized into four distinct types: provided, not provided,
provided too early/too late, provided for too long/too short. For
example, a Control Action turning a cooling fan on may be
unsafe if provided when the object being cooled is already at
its minimum allowed temperature. In this example, the context
is that the object is at a minimum temperature.
Controller constraints are derived from the UCAs, and are
statements that outline the only kind of behavior allowed
by the controller. For example, if a controller can cause a
hazard by issuing a certain command in a specific context,
the Controller Constraints must be specified such that thiscommand cannot be issued in this context. In this work,
rather than outlining each unsafe context in the Controller
Constraints, the only safe context is often specified due to
being more intuitive and efficient in the case of this specific
system.
B. IEC-61499
The IEC 61499 standard provides a means of creating
distributed and intelligent control and automation systems.
Central to the standard is the use of Function Block Diagrams
which consist of interconnected Function Blocks. These Func-
tion Blocks are event-driven and consist of both event and
data inputs and outputs. Data-inputs and outputs are always
associated with the events they are transmitted with. Within
each Function Block is an Execution Control Chart (ECC), a
type of finite-state machine that determines the behavior of the
Function Block. Each state in the ECC defines the algorithms
and output events associated with it. [11]
C. Pick-and-place machine
The pick-and-place manipulator presented in Fig. 2, used as
a simple case study in this paper, is a stationary machine used
to transfer items between locations. The machine used in this
experiment handles a total of 4 locations where items either
appear or disappear at: 3 input trays and 1 output tray. The
objective of the pick-and-place machine is to transfer these
items, referred to as workpieces, from the input trays to the
output tray. The input trays can hold one workpiece each at a
time, and the output tray disposes of the workpieces as they
are placed in it.
The machine consists of 4 actuators: two horizontal linear
actuators, a vertical linear actuator, and a vacuum gripper.
The horizontal actuators are used to navigate between the
trays, while the vertical actuator is used to lower and raise
the vacuum gripper to and from the trays. The output tray is
located such that both horizontal actuators must be retracted
for the vacuum gripper to be maneuvered above it. The
location of input tray 1 corresponds to just the number 1
actuator being fully extended, while the location of input tray
2 corresponds to just the number 2 horizontal actuator being
fully extended. Input tray 3 is reached by fully extending both
the number 1 and 2 horizontal actuators.
The horizontal actuators are controlled using extend and
retract commands to each actuator. The actuators extend/retract
while the commands are given. The vertical actuator is con-
trolled with a single command, extend. When TRUE, the ver-
tical actuator is extended, whereas a FALSE value corresponds
with the vertical actuator being retracted. The state of the
vacuum gripper is toggled using on and off commands.
The pick-and-place machine has inputs corresponding with
each tray, indicating whether or not the vacuum gripper is
above the tray. Similarly, the pick-and-place machine has
inputs indicating if a tray is occupied by an item. There
are also two boolean inputs describing whether the vertical
actuator is fully extended or fully retracted, and an input
describing whether the vacuum is active.

is repeated for any workpieces appearing in the input trays,
and all workpieces must eventually reach the output tray.
D. fbAssistant
The Function Block Assistant is a proof-of-concept tool
developed by Flexbridge AB, which follows an iterative
methodology to develop automation logic with LLM support.
The workflow of the tool is presented in Fig. 3. In the current
work, the tool is mainly used as an interface for OpenAI’s
GPT models and to provide a visual means of checking the
code generated by the LLM. The tool also allows for a very
straightforward process of converting the LLM’s output toexecutable IEC 61499 code on both soft Programmable Logic
Controllers (PLCs) for simulation and real PLCs.
The process begins with the user providing natural-language
descriptions of the required control logic. These descriptions
are then processed by an AI model, which generates a corre-
sponding state machine representation. Once the state machine
is created, it is visualized in the tool for user verification. This
visualization helps identify any missing transitions or incorrect
states. If adjustments are needed, the user refines the textual
input and submits an updated request to the AI.
After validation, the state machine is converted into a
function block that is compatible with industrial automation
tools such as EcoStruxure Automation Expert. The generated
function block is then deployed to a virtual or physical envi-
ronment for testing. Testing involves simulating the generated
control logic within a virtual commissioning environment.
This allows engineers to verify the expected behavior before
deploying it to the actual hardware. The refinement process
ensures that any discrepancies between expected and actual
behavior are resolved prior to full implementation.
Fig. 4 illustrates one step in the state machine refinement
process supported by fbAssistant. Given the incomplete state
machine of the pick-and-place controller on the left, the
user provided five comments in natural language, each of
which was correctly addressed by the assistant, making the
corresponding modifications to the state machine.

with some prompts generating encouraging results, while most
prompts led to dissatisfactory outcomes. Fig. 6 presents a
summary of these results in the form of neutral, negative, and
positive changes introduced by the LLM in each iteration for
each prompt.
The number of overall changes introduced in each set of
iterations and the maximum number of changes introduced
at once vary significantly from one constraint to another,
with some controller constraints generating a maximum of
3 changes in one iteration, while others introduced up to 12

Similarly, while some controller constraints resulted in a

directly reflect a successful or unsuccessful outcome. The
absolute number of changes also does not capture the number
of types of changes introduced. For example, some iterations
introduced up to 12 changes which could be all classified as a
single type of change, while other iterations would introduce
several different types of changes at once.
In many instances, positive changes introduced in later iter-
ations were considered positive due to addressing or reversing
negative changes made in earlier iterations. For example, in
one instance, the value of an output variable in a specific state
was changed back and forth between TRUE and FALSE for
several iterations.
The only set of iterations that could be considered truly
successful was the set that used controller constraint C-7 as
the basis for its prompts. The iterations in this set introduceda total of 10 positive changes, 3 negative changes, and 12
neutral changes. The resulting finite-state machine, presented
in Fig. 7, ensures that the vertical actuator is fully extended
before releasing the workpiece. This is unlike the original
finite-state machine, which dropped the workpiece without
the vertical actuator being extended at all. The best-behaved
versions of the finite-state machine with controller constraint
C-7 were reached in iteration 11. This version also ensured that
the vertical actuator was fully retracted before proceeding to
the ”WAIT” state, essentially preventing the vertical actuator
from being in the extended state when maneuvering to one
of the input trays. The LLM was also hesitant to introduce
changes to this version of the finite-state machine, with no
changes being introduced for the following 5 iterations.
Other sets of iterations failed due to a variety of reasons.
One of the most common and significant negative changes
introduced through the prompts was the removal of the states
describing how to maneuver to each input tray. These changes
often occurred over the course of several iterations, starting
from small changes with one variable, then escalating to the
removal of transitions, and finally the removal of a state. In
many cases, the state describing how to reach the 3rd input
tray would first be altered to mimic the state describing how
to reach the 2nd input tray, and then eventually be removed

B. Second round of iterations
In addition to the 180 iterations conducted in 9 sets of 20
iterations for each prompt separately, 1 set of 10 iterations was
conducted with a prompt including both controller constraints
C-1 and C-2. The number of changes introduced in this set of
iterations was similar to the earlier sets despite the set spanning
only 10 iterations, with 19 positive, 20 negative, and 6 neutral
changes over the course of 10 iterations.
Combining the two controller constraints into one prompt
appears to improve the results. Most notably, new variables
were not introduced, and the iterations did not drift towards
removing the ”EXT H3” state unlike iterations conducted with
each of the controller constraints C-1 and C-2 separately.
V. C ONCLUSIONS
This paper has described and presented results of the first
attempts in evolving finite-state machines with the aid of an
LLM and prompts based on controller constraints derived
from an STPA analysis. While the results shed light on the
characteristics that may be desirable in prompts used to iterate
control software, the prompting method was only applied
successfully in 1 of 10 prompts tested in this paper. Key
findings are summarized below:
•As implemented in this paper, iteratively evolving the
controller with controller constraints often falls short of
the intended outcome.
•The results of each iteration tend to drift/migrate slowly
over the course of several iterations, however, not always
in a positive direction.
•Prompts that define the intended behavior of a specific
actuator poorly or partially appear to leave more room
for the LLM to assume behavior and introduce unwanted
changes into the results. A more complete description
of the intended behavior of the actuator in each prompt
appears to improve results and mitigate some of the
identified issues.
•Controller constraints should likely be formulated such
that they describe the context in terms of the variables
existing in the FSM, in order to limit the generation of
superfluous variables.
•Prompts in this paper were formatted so as to not un-
dermine the ”confidence” of the LLM, as such prompts
have been identified as detrimental to the outcomes of
iterative prompting [8]. Further improving the language
used in the prompt may reduce the number of negative
and superfluous changes made by the LLM.
A key weakness of the approach explored in this paper is
its susceptibility to introducing unwanted drift to the generated
code over the course of several iterations. This may lead to the
correct parts of the existing code to be altered significantly.
These issues could potentially be addressed by improving
prompt formulation and limiting the number of iterations. Us-
ing several controller constraints together or formulating themsuch that they each describe the behavior of the component
being controlled more completely could likely also generate
significant improvements. However, more work is required to
confirm this.
In addition to confirming the impact of combining controller
constraints and formulating constraints that fully describe
behavior of each component, future work should focus on
exploring alternative sources for prompts, for example system
descriptions, and how they could be introduced into iterations.
Additionally, future work should identify the stages of code
generation in which iterating the code with requirements
provides the best results, as the finite-state machine used as
the starting point for each set of iterations was already in a
rather mature stage compared to the true starting point of an
empty FSM.
While the iterative prompting method presented here pro-
vided interesting insight into the behavior of LLMs in the
context of evolving FSMs with an iterative prompting ap-
proach, alternative iterative prompting methods should also be
explored, such as those iterating on the original code compared
to iterating recursively with the previous iteration of code.
