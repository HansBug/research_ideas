## 简报

**解决的问题**：使用LLM和STPA控制器约束迭代优化有限状态机，生成IEC 61499控制代码

- **输入**：
  - 初始有限状态机（FSM）
  - STPA分析生成的控制器约束（Controller Constraints）
  - 自然语言提示词（基于控制器约束）
- **方法**：
  - fbAssistant工具（LLM接口）
  - 迭代提示策略（每个约束迭代20次）
  - GPT模型进行状态机优化
- **输出**：
  - 优化后的有限状态机
  - IEC 61499兼容的控制代码
  - 可部署到PLC的功能块

```

│         ↓                                                        │
```

**实验结果**：在9组提示词测试中，仅1组（约束C-7）成功优化FSM，修复了垂直执行器未完全伸展就释放工件的问题。该组在20次迭代中引入10个正面变更、3个负面变更、12个中性变更，第11次迭代达到最佳状态。其他8组测试因状态漂移、引入新变量、移除关键状态等问题失败。

**研究动机**：LLM在Python等通用编程语言的代码生成中已显示出显著价值，但在工业自动化领域（特别是IEC 61499标准）的应用仍处于早期阶段。现有工具（如fbAssistant）虽然支持LLM生成IEC 61499代码，但需要大量人工干预，且提示词策略尚未明确。迭代提示在其他领域已证明能提高结果质量，但在控制软件生成中的效果未知。此外，缺乏系统化的提示词生成方法，STPA作为成熟的危害分析方法，其生成的控制器约束可为提示词提供结构化来源，并将代码生成整合到系统工程流程中。

**方法创新**：
1. 首次将STPA控制器约束作为LLM迭代提示的系统化来源
2. 探索递归迭代策略（基于前一次迭代结果而非原始FSM）在FSM优化中的应用
3. 提出变更分类评估方法（正面/负面/中性）量化迭代效果
4. 将安全分析（STPA）与LLM代码生成流程整合

**实验设计**：
- **对比对象**：无传统baseline，主要对比不同控制器约束的迭代效果
- **数据集**：pick-and-place机器案例（4个执行器、多个传感器、4个托盘位置）
- **评估指标**：变更数量和类型（正面/负面/中性）、FSM行为正确性、状态漂移程度
- **使用模型**：OpenAI GPT模型（通过fbAssistant工具调用）
- **实验规模**：9个控制器约束 × 20次迭代 + 1组组合约束 × 10次迭代 = 190次迭代

**结论与不足**：
- **主要结论**：迭代提示策略在控制软件生成中可行但成功率低（1/10），约束C-7成功修复了关键安全问题（工件掉落）
- **优势**：成功案例展示了LLM能自动引入安全保障（如确保垂直执行器完全伸展），且在达到最佳状态后保持稳定（5次迭代无变更）
- **局限性**：
  1. 状态漂移严重：FSM在多次迭代中逐渐偏离原始结构，关键状态被移除
  2. 引入不必要变量：LLM创建新变量而非使用现有传感器输入
  3. 过度假设：LLM对未明确定义的行为做出错误假设（如互补变量行为）
  4. 约束描述不完整：部分约束未充分描述执行器完整行为，导致LLM填补空白时引入错误
  5. 语言不一致：约束中的术语与FSM变量名不匹配，降低生成质量

## 研究问题与动机

### 研究背景

大语言模型（LLM）在软件开发中的应用日益广泛，特别是在代码生成领域已展现出显著价值。研究表明，LLM在Python等通用编程语言的代码生成中能带来显著的时间节省。然而，在工业自动化软件开发领域，特别是IEC 61499标准的控制应用中，LLM的应用仍处于早期探索阶段。

IEC 61499是工业自动化领域的重要标准，用于创建分布式智能控制系统。该标准基于功能块图（Function Block Diagrams），其中每个功能块包含执行控制图（ECC），本质上是一种有限状态机（FSM）。虽然已有工具（如fbAssistant）开始支持使用LLM生成IEC 61499代码，但这些工具仍需要大量人工干预才能生成功能性的控制软件。

### 核心问题

本研究要解决的核心问题包括：

1. **提示词策略不明确**：当前使用LLM生成IEC 61499控制软件时，什么类型的提示词能产生最佳结果尚不清楚，最合适的提示策略也未被识别。

2. **迭代优化效果未知**：迭代提示策略在其他领域（如提高回答真实性）已显示出改进效果，但在控制软件生成中是否同样有效尚未研究。特别是，使用相同提示词反复迭代软件是否能使控制软件收敛到更符合预期行为的解决方案？

3. **缺乏系统化的提示词来源**：即使迭代策略有效，也需要一个系统化的方法来生成高质量的提示词，以有效利用潜在的收敛特性。

4. **与系统工程流程脱节**：现有的LLM代码生成方法缺乏与更广泛的系统工程工作流程的整合。

### 研究动机

**STPA（System-Theoretic Process Analysis）作为解决方案**：STPA是一种成熟的危害分析方法，能够识别设计缺陷和系统组件交互产生的危害。STPA分析的第三步会生成不安全控制动作（UCAs）和控制器约束（Controller Constraints）。这些控制器约束明确定义了控制器允许的唯一行为类型，为LLM提示词提供了结构化、系统化的来源。

使用STPA控制器约束作为提示词来源具有以下优势：
- 提供明确定义的安全约束
- 与系统工程流程自然整合
- 基于系统化的危害分析，而非临时的需求描述
- 能够追溯到具体的安全目标和危害场景

### 研究意义

如果迭代提示策略能够使FSM收敛到更好的解决方案，结合STPA提供的系统化提示词来源，这种方法可以：
- 以极少或无需人工干预的方式生成功能性控制软件
- 将控制软件生成整合到系统工程流程中
- 提高控制工程师的生产力
- 为工业自动化领域的LLM应用提供实践指导

## 核心方法

### 方法概述

本研究提出了一种基于STPA控制器约束的LLM迭代提示方法，用于优化有限状态机并生成IEC 61499代码。整个方法包括三个主要阶段：

1. **STPA分析阶段**：对目标系统进行系统理论过程分析，生成控制器约束
2. **迭代优化阶段**：使用控制器约束作为提示词，通过LLM递归迭代优化FSM
3. **代码生成与部署阶段**：将优化后的FSM转换为IEC 61499功能块代码并部署

### STPA分析方法

STPA方法包含4个步骤，本研究使用前3步：

**步骤1：确定分析目的**
- 定义损失（Losses）：需要避免的结果（如人员伤亡、设备损坏、任务失败）
- 定义危害（Hazards）：可能导致损失的系统状态
- 定义系统约束（System Constraints）：否定危害的陈述
- 确定系统边界

**步骤2：建模控制结构**
- 开发控制结构图（Control Structure Diagram）
- 描述控制器与被控过程之间的关系
- 通过控制动作（Control Actions）和反馈建立层次化控制关系
- 可在不同抽象层次建模

**步骤3：识别不安全控制动作和控制器约束**
- 生成不安全控制动作（UCAs）：描述控制动作在何种情况下变得不安全
- UCA分为四类：提供、未提供、提供过早/过晚、提供时间过长/过短
- 从UCAs推导控制器约束：明确定义控制器允许的唯一行为
- 控制器约束指定安全上下文，而非列举所有不安全上下文

### fbAssistant工具

fbAssistant是由Flexbridge AB开发的概念验证工具，用于通过LLM支持迭代开发自动化逻辑。工具工作流程：

1. **自然语言输入**：用户提供控制逻辑的自然语言描述
2. **AI模型处理**：AI模型生成相应的状态机表示
3. **可视化验证**：在工具中可视化状态机，识别缺失的迁移或错误的状态
4. **迭代优化**：用户优化文本输入并提交更新请求给AI
5. **代码生成**：将验证后的状态机转换为功能块，兼容EcoStruxure Automation Expert等工业自动化工具
6. **测试部署**：部署到虚拟或物理环境进行测试

本研究主要使用fbAssistant作为OpenAI GPT模型的接口，以及提供可视化检查生成代码的手段。

### 迭代提示策略

**递归迭代方法**：
- 每个提示词应用于前一次迭代的结果，而非原始FSM
- 每个控制器约束作为独立提示词进行20次迭代
- 提示词格式："If the following constraints do not hold for the state machine, modify it such that the constraints hold: [控制器约束]"

**变更跟踪与评估**：
- 记录每次迭代引入的所有变更
- 将变更分类为：
  - **正面变更**：修复问题（如工件掉落、语法错误）
  - **负面变更**：引入不期望的行为或错误语法
  - **中性变更**：对行为影响很小或无影响的变更（如不必要的变量定义）

**假设验证**：
基于初始180次迭代的观察，提出并测试两个假设：
1. 使用与现有控制软件变量不一致的语言可能降低生成代码质量
2. 每个提示词应完整描述被控执行器或设备的行为，以限制LLM做出不良假设

验证方法：将控制器约束C-1和C-2组合成单个提示词，并重写约束使用实际输入变量（s0, s1, s2, s3）而非抽象术语（"above an input tray"）。

### 案例系统：Pick-and-Place机器

**系统描述**：
- 功能：在输入托盘和输出托盘之间转移工件
- 托盘配置：3个输入托盘 + 1个输出托盘
- 执行器：2个水平线性执行器、1个垂直线性执行器、1个真空夹具
- 传感器：托盘位置传感器、托盘占用传感器、垂直执行器位置传感器、真空状态传感器

**预期行为序列**：
1. 工件到达输入托盘
2. 将真空夹具移动到输入托盘上方
3. 降低真空夹具并打开真空
4. 抬起真空夹具（携带工件）
5. 将工件移动到输出托盘上方
6. 降低工件到输出托盘
7. 关闭真空
8. 抬起真空夹具
9. 重复处理其他工件

**STPA分析结果**：
- 4个损失（Losses）
- 4个危害（Hazards）
- 33个不安全控制动作（UCAs）
- 11个控制器约束（C-1至C-11）

### 实验设计

**初始轮次**：
- 9个控制器约束（C-1, C-2, C-3, C-4, C-7, C-8, C-9, C-10, C-11）
- 每个约束独立进行20次迭代
- 总计180次迭代
- 从相同的初始FSM开始

**第二轮次**：
- 组合约束C-1和C-2
- 重写约束使用实际变量名
- 进行10次迭代
- 验证假设

**评估方法**：
- 主观评估每次迭代的变更类型和数量
- 分析FSM行为的正确性
- 识别状态漂移和结构变化
- 记录语法错误和不可解析的输出

## 实验与评估

### 初始轮次实验结果

**整体表现**：
- 9组提示词产生了差异显著的结果
- 仅1组（约束C-7）被认为真正成功
- 不同约束引入的变更数量差异巨大：从20个到69个变更不等
- 单次迭代最多引入12个变更

**成功案例：约束C-7**

约束C-7要求："控制器必须仅在拾取或放置序列中提供vextend命令"

**成果**：
- 总变更：10个正面 + 3个负面 + 12个中性
- 第11次迭代达到最佳状态
- 修复了关键安全问题：确保垂直执行器在释放工件前完全伸展（原始FSM在未伸展时就掉落工件）
- 额外改进：确保垂直执行器在移动到输入托盘前完全收回
- 稳定性：达到最佳状态后，连续5次迭代无变更，显示LLM对该版本的"信心"

**失败模式分析**：

1. **状态漂移/迁移**（最常见且严重）
   - 描述：FSM在多次迭代中逐渐偏离原始结构
   - 典型过程：
     - 小变更开始（修改单个变量）
     - 逐步升级（移除迁移）
     - 最终移除整个状态
   - 具体案例：描述如何到达第3个输入托盘的状态（EXT_H3）首先被修改为模仿第2个托盘的状态（EXT_H2），然后被完全移除
   - 有趣现象：某些情况下，被断开连接的状态在几次迭代后被重新引入，但结构不同

2. **引入不必要的新变量**
   - 约束C-1和C-2都引入了新输入变量"aboveTray"和"vacantTray"
   - 这些变量对应约束中的上下文描述
   - 应该使用现有传感器输入（s0, s1, s2, s3）来定义
   - 特殊案例：约束C-9引入了新输出变量"vextend_stop"，虽然未明确要求，但LLM正确地将其添加到初始化状态

3. **过度假设互补行为**
   - LLM经常假设互补变量的行为
   - 示例：如果约束定义在特定情况下提供"打开真空"命令，LLM会假设在所有其他情况下应提供"关闭真空"命令
   - 实际上：正确行为包括许多状态中两个命令都不应提供

4. **错误的迁移条件**
   - 某些迭代引入了依赖输出变量而非输入变量的迁移条件
   - 示例：在命令打开真空的状态中，迁移条件检查输出变量"turn_vacuum_on"而非输入变量"vacuum"
   - 结果：迁移条件实际上什么都不检查，而正确的条件应检查真空的实际状态

5. **无关变更**
   - 某些迭代修改了与约束无关的FSM方面
   - 示例：约束C-8（关于垂直执行器）的迭代多次修改"turn_vacuum_off"命令的值，尽管约束中没有提及真空夹具

6. **语法错误和解析失败**
   - 4次返回不可解析的JSON文件（重新运行后都成功）
   - 少数情况下产生语法错误（缺少空格、错误符号）
   - 语法错误从未持续超过3次迭代

### 第二轮次实验结果

**组合约束C-1和C-2**：
- 10次迭代产生：19个正面 + 20个负面 + 6个中性变更
- 尽管迭代次数减半，变更数量与初始轮次相似

**改进效果**：
- 未引入新变量（验证了假设1）
- 未出现向移除"EXT_H3"状态漂移的趋势（验证了假设2）
- 表明组合约束并使用一致的变量名确实改善了结果

### 关键发现

1. **迭代策略的局限性**：
   - 递归迭代方法在9/10的提示词测试中未达到预期结果
   - 状态漂移是主要障碍，可能导致正确部分被显著改变

2. **提示词质量的重要性**：
   - 不完整或部分描述执行器行为的提示词给LLM留下了假设空间
   - 更完整的行为描述能改善结果并缓解已识别的问题

3. **语言一致性的影响**：
   - 约束应使用FSM中存在的变量来描述上下文
   - 避免抽象术语可以限制生成多余变量

4. **提示词信心的影响**：
   - 提示词格式设计为不破坏LLM的"信心"
   - 进一步改进提示词语言可能减少负面和多余变更

5. **组合约束的优势**：
   - 同时使用多个约束或使其更完整地描述组件行为可能产生显著改进
   - 需要更多工作来确认这一点

- 4个危害定义
- 33个不安全控制动作
- 11个控制器约束
- 初始FSM包含多个状态（WAIT, EXT_H1, EXT_H2, EXT_H3, EXT_VACUUM, 等）

**评估指标**：
- 变更数量（正面/负面/中性）
- 变更类型的多样性
- FSM行为正确性（是否修复了工件掉落等问题）
- 状态结构稳定性（是否发生漂移）
- 语法正确性
- 变量使用合理性

### 主要实验结果总结

**成功率**：1/10（10%）

**最佳结果**：
- 约束C-7在第11次迭代达到最佳状态
- 修复了垂直执行器未完全伸展就释放工件的关键安全问题
- 额外引入了垂直执行器在水平移动前完全收回的保障

**失败原因分布**：
- 状态漂移：最常见，导致关键状态被移除
- 引入新变量：降低了代码可用性
- 过度假设：引入错误的互补行为
- 约束不完整：给LLM留下过多假设空间

**改进方向验证**：
- 组合约束：有效
- 使用一致变量名：有效
- 完整描述行为：理论上有效，需进一步验证

### 相关性分析

本论文与博士研究的核心主题"基于LLM的控制系统状态机建模与验证"高度相关，特别是在以下方面：

1. **研究内容一（LLM状态机建模）的直接相关性**：
   - 本论文探索了使用LLM生成和优化有限状态机的方法
   - 提出了基于STPA约束的迭代提示策略
   - 揭示了LLM在FSM生成中的具体挑战（状态漂移、过度假设等）

2. **验证驱动的建模思路**：
   - STPA控制器约束本质上是安全性质的形式化表达
   - 这与研究内容二（验证场景与性质生成）和研究内容三（基于验证剖面的验证）的思路一致
   - 展示了如何将安全约束整合到生成流程中

3. **迭代修复方法的启发**：
   - 论文的递归迭代策略与研究内容四（迭代式模型修复）直接相关
   - 虽然成功率低，但成功案例（C-7）展示了迭代修复的潜力
   - 失败模式分析为改进迭代策略提供了重要洞察

4. **工业标准的实践**：
   - IEC 61499是工业自动化领域的重要标准
   - 与博士研究关注的控制系统软件需求场景一致
   - 提供了实际工业应用的参考案例

### 可借鉴之处

1. **STPA作为需求来源的系统化方法**：
   - STPA提供了从危害分析到控制器约束的系统化流程
   - 可以借鉴这种方法，将安全分析整合到LLM建模流程中
   - 控制器约束可以作为待验证性质的来源

2. **迭代提示策略的设计经验**：
   - 提示词应完整描述组件行为，避免给LLM留下假设空间
   - 提示词应使用与现有模型一致的术语和变量名
   - 组合多个相关约束可能比单独使用更有效
   - 提示词格式应保持LLM的"信心"，避免过度质疑

3. **变更分类评估方法**：
   - 将变更分为正面/负面/中性是一种实用的评估方法
   - 可以借鉴这种方法来评估迭代修复的效果
   - 有助于量化和追踪模型质量的演变

4. **失败模式的识别**：
   - 状态漂移、过度假设、引入新变量等失败模式具有普遍性
   - 可以设计针对性的检测和预防机制
   - 为改进LLM提示策略提供了具体方向

5. **工具链整合思路**：
   - fbAssistant展示了如何将LLM整合到工业自动化工具链中
   - 可视化验证、代码生成、部署测试的完整流程值得借鉴
   - 为构建端到端的LLM建模工具提供了参考架构

### 存在的不足与改进空间

1. **成功率过低**：
   - 10%的成功率在实际应用中不可接受
   - 需要更有效的提示策略或约束表达方式
   - **改进方向**：可以引入形式化的约束表达（如时序逻辑），而非自然语言描述

2. **缺乏自动化验证**：
   - 论文依赖主观评估判断变更的正面/负面性质
   - 没有使用模型检查等形式化验证方法
   - **改进方向**：整合UPPAAL等验证工具，自动检测每次迭代是否满足安全性质

3. **状态漂移问题严重**：
   - 递归迭代容易导致FSM逐渐偏离原始结构
   - 缺乏保持结构稳定性的机制
   - **改进方向**：引入结构约束或锚点状态，限制LLM的修改范围；或采用非递归迭代策略

4. **约束表达的局限性**：
   - 自然语言约束容易产生歧义
   - LLM对约束的理解可能不准确
   - **改进方向**：使用结构化的约束模板或领域特定语言（DSL）

5. **缺乏理论分析**：
   - 论文主要基于实验观察，缺乏理论解释
   - 未分析为什么某些约束成功而其他失败
   - **改进方向**：分析约束的形式化特征与成功率的关系

6. **单一案例的局限性**：
   - 仅使用pick-and-place机器一个案例
   - 结论的普遍性有待验证
   - **改进方向**：在多个控制系统案例上进行实验

7. **未考虑时间约束**：
   - STPA约束主要关注逻辑正确性
   - 未涉及时间自动机的时间约束
   - **改进方向**：扩展方法以支持时间约束的生成和验证

8. **LLM能力的依赖**：
   - 方法的效果高度依赖LLM的能力
   - 未探索不同LLM模型的表现差异
   - **改进方向**：测试不同模型（如Claude、GPT-4等）的表现

### 对本研究的启发

1. **验证驱动的生成方法**：
   - 可以将STPA的思路扩展为"验证驱动的LLM建模"
   - 在生成过程中持续验证安全性质
   - 使用验证反馈指导LLM的迭代优化

2. **约束的形式化表达**：
   - 研究如何将自然语言约束转换为形式化性质（LTL/CTL）
   - 探索形式化约束作为LLM提示的效果
   - 建立约束表达与验证性质之间的映射

3. **迭代修复的改进策略**：
   - 设计非递归迭代策略，避免状态漂移
   - 引入结构保持机制，限制修改范围
   - 使用验证结果（反例）作为迭代的输入

4. **多层次的提示策略**：
   - 区分结构层面的提示（状态、迁移）和行为层面的提示（守卫、动作）
   - 分阶段生成：先生成结构，再优化行为
   - 使用层次化的约束组织方式

5. **评估方法的完善**：
   - 结合主观评估和形式化验证
   - 定义量化指标（如结构相似度、性质满足率）
   - 建立基准数据集和评估框架

6. **工具链的构建**：
   - 借鉴fbAssistant的架构，构建LLM建模工具
   - 整合STPA分析、LLM生成、形式化验证、迭代修复的完整流程
   - 提供可视化界面和交互式优化功能
