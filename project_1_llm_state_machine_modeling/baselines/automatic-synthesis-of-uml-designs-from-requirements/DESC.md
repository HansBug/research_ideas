## 简报

本文关注的问题是：在高度迭代的 UML 开发流程中，如何把 use cases / scenarios 自动推向 class diagram 与 statecharts 等设计工件。

- **输入**：用例、场景与 sequence diagrams
- **方法**：从场景化需求出发，综合生成 UML 设计并在迭代修改过程中保持 requirements-to-design 一致性。
- **输出**：UML 设计工件（含 statecharts）

```text
用例/场景 -> sequence diagrams -> UML 设计综合 -> statecharts/类图
```

一句话结论：说明了从需求场景向状态图设计自动过渡的早期思路，对“需求到行为设计”链路有历史参考价值。

## 研究问题与动机

### 问题背景
在高度迭代的 UML 开发流程中，如何把 use cases / scenarios 自动推向 class diagram 与 statecharts 等设计工件。

### 核心问题
作者试图回答：在给定 用例、场景与 sequence diagrams 的前提下，怎样更稳定地得到 UML 设计工件（含 statecharts），或至少把需求加工为更接近该目标的形式化中间层。

核心动机是减少需求到行为设计之间的人工鸿沟，提高可追溯性、一致性以及后续验证/执行的可行性。
