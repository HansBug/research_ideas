# `baselines/` 操作规范

本文档用于固定 `project_1_llm_state_machine_modeling/baselines/` 与 [SUMMARY.md](./SUMMARY.md) 的后续维护方式，作为 `project_1` 中 baseline 论文持续收集、整理、比较与回填的统一工作规范。

## 0. 文档关系与使用顺序

`baselines/` 下几个核心文档的职责如下：

1. [README.md](./README.md)
   - 解释论文集定位、纳入/排除标准与单论文目录要求。
   - 是进入本论文集时的首个入口文档。
2. [GUIDE.md](./GUIDE.md)
   - 规定检索、筛选、目录维护、`DESC.md` 生产与 [SUMMARY.md](./SUMMARY.md) 回填规范。
3. [SUMMARY.md](./SUMMARY.md)
   - 记录当前正式收录范围、统计口径、论文清单、数据集/benchmark 盘点、待补充候选与更新日志。
4. [DESC_GUIDE.md](./DESC_GUIDE.md)
   - 规定单篇 `DESC.md` 的写法、章节结构、比较维度和证据要求。

默认推荐顺序如下：

1. 先读 [README.md](./README.md)，理解收录边界。
2. 再读 [GUIDE.md](./GUIDE.md)，确认本轮工作流程。
3. 再读 [SUMMARY.md](./SUMMARY.md)，确认当前总账和待补充缺口。
4. 若任务涉及单篇分析，再读 [DESC_GUIDE.md](./DESC_GUIDE.md)。
5. 最后进入具体论文目录，按 `bibtex.bib -> paper_content.txt -> paper.pdf（必要时） -> DESC.md` 的顺序工作。

## 1. 目标与任务边界

`baselines/` 不是一般性的“相关工作收纳箱”，而是为 `project_1` 建设可反复引用的 LLM 建模 baseline 集合。它主要服务以下任务：

1. 固定“可与本研究横向比较”的论文集。
2. 提取统一可比字段，为后续论文写作和实验设计服务。
3. 沉淀对控制系统状态机建模真正有指导价值的研究观察和空白。

以下工作不属于本论文集的主任务，应避免混入：

1. `sources/` 中那类控制系统样本论文的收集与 `STM.md` 提取。
2. 泛形式化方法背景收集、传统方法综述、非 LLM 工具普查。
3. FSM 只作为提示范式、规划控制或多智能体编排机制的论文。

## 2. 检索策略

### 2.1 主线关键词簇

基于 `BASELINE.md`、`BASELINE_TO_EXPLORE.md`、`DIR_1_llm_direct_state_machine_generation.md` 和 `LITERATURE_SEARCH_RESULTS.md` 的现有经验，后续检索应优先围绕以下主线：

1. `LLM/GPT/ChatGPT/Claude/Gemini` + `state machine/statechart/state diagram/FSM`
2. `LLM/GPT` + `SysML/UML/behavior model/domain model`
3. `LLM` + `model refinement/model completion/model repair/test generation`
4. `multimodal/diagram recognition/vision-language` + `state diagram/FSM`
5. `requirements` + `statechart/state machine` + `automotive/industrial/control`

### 2.2 检索方向优先级

优先级按以下顺序执行：

1. 直接生成状态机或行为模型的论文。
2. 基于已有状态图/FSM 的精化、扩展、修复、测试生成论文。
3. 泛 UML/SysML 生成论文，但需能补强 `project_1` 的方法比较维度。
4. 邻近任务论文，例如状态图识别到代码、FSM 推理 benchmark，仅在其方法对本研究有直接借鉴价值时纳入。

### 2.3 去偏与跳过规则

为避免把时间耗在低命中方向，后续应主动降权以下候选：

1. `FSM` 仅用于智能体控制、规划或对话流，而非建模工件生成。
2. 纯机器人操作流水线、纯协议推理、纯智能合约生成等与 `project_1` 对比关系较弱的方向。
3. 纯综述、纯位置论文、纯经验报告，若缺少可比较实验与方法细节。

### 2.4 批量规模

本论文集领域相对窄，默认采用以下 override：

1. 每轮实际筛查的候选论文尽量不少于 `12` 篇。
2. 每轮最终正式入库若少于 `2` 篇，需要在 [SUMMARY.md](./SUMMARY.md) 的更新日志中说明原因。

## 3. 筛选标准

### 3.1 收录条件

一篇论文进入 `baselines/`，至少应满足以下条件：

1. 论文核心任务与 LLM 建模工件生成、精化、扩展、修复或识别直接相关。
2. 输入、输出、方法流程、实验设置和主要结果可从原文稳定提取。
3. 对 `project_1` 的研究设计、baseline 选择或比较维度有明确价值。
4. PDF 原文和 `paper_content.txt` 可获得并质量可用。

### 3.2 降优先级条件

以下论文可暂存为候选，但不应优先入库：

1. 任务更偏代码生成、协议推理或图样识别，只有一部分可迁移到 `project_1`。
2. 输出不是状态机/UML/SysML 模型，但方法思路值得对照。
3. 原文信息不完整，例如公开性、评测细节或输入输出定义不清。

### 3.3 排除条件

以下论文原则上不纳入：

1. 非 LLM 方法，且仅作为背景对比。
2. 不产生可比较建模工件，或 `FSM` 只是内部控制结构。
3. 只有抽象观点，没有足够实验或方法证据。

### 3.4 去重规则

去重按以下优先级执行：

1. 优先按 DOI 去重。
2. DOI 缺失时按标准化标题去重。
3. 标题存在轻微差异时，再结合作者、年份和会议/期刊综合判断。

### 3.5 BASELINE评估口径

为避免把“相关工作”与“可直接对比 baseline”混在一起，`baselines/` 中每篇正式收录论文都必须给出一个 `BASELINE评估` emoji，统一使用以下口径：

| Emoji | 含义 |
|---|---|
| 🟢 | 直接构成 baseline：输入是自然语言系统设计/描述/需求，允许附带轻量补充规格（如 I/O 接口、角色说明、示例）；输出是状态机、Statechart、SysML 行为模型或与本研究目标高度等价的行为模型，与 `project_1` 的“自然语言自动生成状态机模型”任务直接可比 |
| 🟡 | 相关但非直接 baseline：与状态机生成、精化、扩展、修复、多模态识别、测试生成密切相关，但输入不是纯自然语言，或任务本质是已有模型精化/扩展，而非从零建模 |
| 🟠 | 弱相关/可比性有限：与有限状态推理、泛 UML/SysML 建模或邻近代码生成有关，但与本研究任务不构成公平直接对比，只适合补充说明或方法借鉴 |
| ⚪ | 背景资料：只适合当背景文献，不建议纳入 baseline 对比表 |

## 4. 目录与文件规范

`baselines/` 下每篇论文必须独占一个子目录。目录名应保持简洁、稳定、可读，通常用标题关键词或已有 slug。

每个单论文目录至少应包含：

1. `paper.pdf`
2. `paper_content.txt`
3. `bibtex.bib`
4. `DESC.md`

其中：

1. `paper_content.txt` 必须由 PDF 自动提取获得。
2. `DESC.md` 是本论文集的核心派生文件，统一遵循 [DESC_GUIDE.md](./DESC_GUIDE.md)。
3. 若目录中已有代码仓库、数据集链接或补充材料信息，应优先写入 `DESC.md`，而不是只留在零散笔记里。

## 5. 内容整理策略

### 5.1 单篇论文需要提取什么

每篇 baseline 的 `DESC.md` 至少应抽取以下可比较信息：

1. 输入工件、输入形态和输入模态。
2. 输出工件、输出模型类型和最终产物形态。
3. 所用 LLM、提示工程、反馈机制、验证或修复闭环。
4. 数据集/benchmark 来源、规模、构造方式、开放性。
5. 主要实验结果、局限性和与 `project_1` 的关系。
6. `BASELINE评估` emoji 与评估理由。

这里必须明确区分：

1. 输入是否是纯自然语言。
2. 若不是，额外输入到底是已有模型、图像、代码、约束、标准文档还是其他工件。
3. 输出到底是状态机模型、UML/SysML 模型、文本化建模语言、代码、测试工件，还是这些的组合。

### 5.2 必须进入汇总表的字段

后续写 [SUMMARY.md](./SUMMARY.md) 的论文总表时，每篇论文必须至少包含以下列：

1. `BASELINE评估`
2. `类别`
3. `标题`
4. `年份`
5. `输入`
6. `输出`
7. `输出模型类型`
8. `使用的LLM`
9. `主要方法`
10. `目录`

其中：

1. `输入` 不得只写“文本”或“模型”，要尽量写清是“自然语言需求”“自然语言 + I/O 规格”“初始 FSM + 约束”“状态图图像”等。
2. `输出` 不得只写“模型”，要尽量写清是“SysML 状态机”“Mermaid Statechart”“Umple 状态机”“SystemVerilog FSM 代码”“扩展状态图 + 测试用例”等。
3. `主要方法` 只用一句话概括核心 pipeline，避免写成长段。

### 5.3 数据集/benchmark 汇总表字段

后续 [SUMMARY.md](./SUMMARY.md) 中还必须维护单独的“数据集与 benchmark 清单”表。每条至少包含：

1. `论文`
2. `BASELINE评估`
3. `数据集/Benchmark`
4. `来源类型`
5. `制作方法`
6. `输入`
7. `输出`
8. `规模`
9. `可获取性`
10. `获取方式/链接`
11. `简述来源`

其中：

1. `可获取性` 统一只用 emoji 表示，不写自由文本，固定口径如下：

| Emoji | 含义 |
|---|---|
| 🟢 | 可直接获取：论文给出了可直接下载、浏览或通过清晰公开渠道直接取得完整内容的链接/入口 |
| 🟡 | 需联系申请：论文明确说明需要联系作者团队、项目方或维护方申请获取 |
| 🟠 | 信息不清：论文只提到制作方式、来源或使用过该数据，但没有给出足够清晰的获取路径 |
| 🔒 | 难以取得：数据依赖企业/团队内部资料、付费标准、受限工业资产或其他现实上很难取得的来源 |

2. 这四类不是偏序等级，不应用“比谁更开放”去硬排；它们只是四种不同的获取状态。
3. `获取方式/链接` 一列只要原文或作者工件中存在明确 URL，就必须写成 Markdown 链接。
4. 若只有仓库、Zenodo、项目页、补充材料页等间接入口，也应优先给出该入口链接，而不是只写“见 DESC”。
5. 若没有清晰链接，必须明确写“原文未提供公开下载链接”或等价表述，不能留空。

### 5.4 什么内容进入 `DESC.md`

应写入 `DESC.md` 的内容：

1. 对单篇论文本身可追溯的事实和分析。
2. 后续论文写作时需要直接引用的关键信息。
3. 原文支持的相关工作脉络与论文定位。

不应塞进 `DESC.md` 的内容：

1. 纯粹跨论文的关键词簇经验。
2. 大量尚未核实的候选论文名单。
3. 只有 collection 维度才有意义的统计和宏观观察。

这些内容应进入 [SUMMARY.md](./SUMMARY.md)。

### 5.5 `DESC.md` 写作硬约束

虽然单篇写法的详细模板由 [DESC_GUIDE.md](./DESC_GUIDE.md) 规定，但本 collection 还额外固定以下硬约束；后续生成或重写任何 `DESC.md` 时都必须满足。

先决阅读顺序固定为：

1. 先读 `bibtex.bib`，确认标题、作者、年份、会议/期刊、DOI、链接等元信息。
2. 再尽可能完整阅读 `paper_content.txt`，至少覆盖摘要、引言、方法、实验/案例、结论、相关工作/参考文献。
3. 只有在提取文本存在歧义、图表缺失、公式断裂或版式问题时，才回到 `paper.pdf` 核对。

禁止的做法包括：

1. 只看标题、摘要或关键词就直接写 `DESC.md`。
2. 只在 `paper_content.txt` 中搜索几个命中词后就跳过其余正文。
3. 不读取 `bibtex.bib`，直接凭 PDF 首页手动拼元信息。
4. 没有生成 `paper_content.txt` 就直接从 PDF 零散摘录。

### 5.6 `DESC.md` 必备章节与信息

每篇 `DESC.md` 至少应包含以下部分，并与 [DESC_GUIDE.md](./DESC_GUIDE.md) 的固定顺序保持一致：

1. `基本信息`
2. `简报`
3. `研究问题与动机`
4. `核心方法`
5. `实验与评估`
6. `与本研究的关系`
7. `重要的相关工作`
8. `文献分类总结`

其中有几项是本 collection 的硬要求：

1. `基本信息` 中除了常规元信息，还必须明确写出：
   - 代码/仓库获取方式。
   - 数据集获取方式。
   - 若原文未提供公开入口，必须如实写“原文未提供公开代码/数据集获取链接”，并补充当前可行获取方式。
2. `简报` 必须紧跟在 `基本信息` 之后，必须回答五个问题：
   - 解决的问题。
   - 研究动机。
   - 方法创新。
   - 实验设计。
   - 结论与不足。
3. `简报` 中的“解决的问题”必须同时说清输入、方法、输出，并附字符示意图，展示 `输入层 -> 方法层 -> 输出层` 的完整黑箱流程。
4. `与本研究的关系` 中必须给出该论文的 `BASELINE评估` emoji，并解释为什么属于该类。
5. `实验与评估` 中若出现数据集、benchmark、代码仓库、补充材料链接，必须与 [SUMMARY.md](./SUMMARY.md) 的汇总口径保持一致。

### 5.7 `重要的相关工作` 的分类提取规则

`重要的相关工作` 是 `DESC.md` 中的核心章节，必须从论文原文的 related work、方法、实验和 references 中提取，不能编造或凭印象补齐。

统一按以下五类组织：

1. `重要的前身类工作`
   - 本论文明确基于其继续发展、扩展或改进的工作。
   - 包括作者团队自己的前期工作，也包括原文明说 `build upon / based on / extending / improving upon / inspired by` 的外部工作。
2. `直接参与实验的 baseline`
   - 在实验中被明确当作 baseline 或对比对象，并出现具体结果对照。
3. `提供了重要论证的工作`
   - 用于支撑研究动机、问题定义、现有不足或研究必要性的工作。
4. `在技术上提供了支持的工作`
   - 被论文直接采用、受其启发或作为技术组成部分使用的方法、框架、模型、工具。
5. `其他重要工作`
   - 提供背景知识、理论基础、评估方法或标准支撑，但不直接参与本论文的方法或实验。

每条相关工作至少要写清：

1. 基本信息：作者、年份、标题、会议/期刊。
2. 主要内容：研究对象、方法、关键特点。
3. 原文中的引用证据：章节、表格或尽可能具体的行号/位置。
4. 主要发现或局限性：只在原文有依据时填写。
5. 与本论文的关系：说明它在本论文里扮演什么角色。
6. 佐证内容/技术使用/作用：按所属类别说明它如何支撑当前论文。

编写要求固定如下：

1. 必须基于原文，不能编造未出现的关系或评价。
2. 分类按“在本论文中实际起到的作用”判定，而不是按我们主观觉得它重要。
3. 若同一篇文献横跨多个类别，应在主类别详细写，在其他类别简短交叉引用，避免重复。
4. 第 1-4 类通常详细写，第 5 类可相对简写，但仍要保留可追溯证据。
5. 所有内容统一用中文写，保持客观、准确、可回溯。

## 6. `SUMMARY.md` 撰写规范

[SUMMARY.md](./SUMMARY.md) 必须稳定维护以下章节：

1. 文档定位与使用方式。
2. 当前收录统计。
3. `BASELINE评估` emoji 口径。
4. 检索关键词簇。
5. 论文清单。
6. 数据集与 benchmark 清单。
7. 初步归类/覆盖盘点。
8. 待补充候选或必要阻塞。
9. 更新日志。

### 6.1 章节顺序与标题固定

后续维护 [SUMMARY.md](./SUMMARY.md) 时，默认应尽量保持与当前版本一致的章节顺序，不随每轮工作任意改名或重排。推荐固定为：

1. 文档定位说明 + 推荐使用顺序。
2. `## 当前收录统计`
3. `## BASELINE评估口径`
4. `## 检索关键词簇`
5. `## 论文清单`
6. `## 数据集与 Benchmark 清单`
7. `## 初步归类与覆盖盘点`
8. `## 待补充高优先级候选`
9. `## 更新日志`

其中：

1. 若未来确有必要新增章节，应优先插入到相邻语义区块中，而不是打乱上述主骨架。
2. 除非 collection 目标发生明显变化，否则不建议把“论文清单”和“数据集与 Benchmark 清单”拆成多套并列表。
3. “待补充高优先级候选”可以按实际情况改为“待补充候选/阻塞记录”，但应保持它位于“覆盖盘点”之后、“更新日志”之前。

### 6.2 开头说明的写法

`SUMMARY.md` 开头不应只是标题，建议保持如下功能：

1. 用一小段说明本文件是该 collection 的总账。
2. 明确它负责记录哪些内容，例如 baseline 论文、比较口径、数据集/benchmark 盘点、待补充候选和更新日志。
3. 紧接着给出推荐使用顺序，至少包含：
   - 先读 [README.md](./README.md)
   - 再读 [GUIDE.md](./GUIDE.md)
   - 再读本文件
   - 如需重写单篇分析，再读 [DESC_GUIDE.md](./DESC_GUIDE.md)

### 6.3 当前收录统计的写法

`## 当前收录统计` 默认使用短列表，不写成长段。至少应包含：

1. 已收录 baseline 论文数。
2. 本轮新增论文数。
3. 已完成 `DESC.md` 数。
4. `⏳ 尚未提取` 数。
5. 若本轮有命名规范化、字段口径变更、结构调整等维护动作，可加一条“本轮规范化工作”。

统计规则如下：

1. 所有数字必须与当前目录和表格真实内容一致。
2. 如果本轮没有新增论文，但进行了结构整理，也应如实写 `0`，不要省略。
3. `DESC.md` 的数量应按当前目录中已完成且正式入账的单篇分析计算。

### 6.4 `BASELINE评估口径` 的写法

`## BASELINE评估口径` 默认使用 emoji 对照表，不改成自然段说明。表格至少包含两列：

1. `Emoji`
2. `含义`

要求如下：

1. 与第 `3.5` 节完全保持一致，不允许在 [SUMMARY.md](./SUMMARY.md) 内另发明一套不同定义。
2. 这里写的是“collection 级比较口径”，不在这一节夹带单篇论文评价。

### 6.5 `检索关键词簇` 的写法

`## 检索关键词簇` 默认拆成以下四个三级小节：

1. `### 当前推荐关键词簇`
2. `### 已观察到的高命中特征`
3. `### 已观察到的低命中特征`
4. `### 检索倾向调整`

写法要求如下：

1. 每个小节优先使用短 bullet 列表。
2. 每节原则上不超过 `10` 行，只保留当前最有指导意义的经验。
3. 更新方式应是整合改写，不是每轮机械追加旧观察。
4. 关键词簇要偏向可执行检索语义，如“模型名/任务词/领域词”的组合，而不是零碎长句。

### 6.6 `论文清单` 的写法

`## 论文清单` 必须使用统一表格，不应在表格外再额外维护一套重复清单。固定列顺序如下：

1. `#`
2. `评估`
3. `类别`
4. `标题`
5. `年份`
6. `输入`
7. `输出`
8. `输出模型类型`
9. `使用的LLM`
10. `主要方法`
11. `目录`

表格写法要求如下：

1. `评估` 列直接使用 emoji，不重复文字标签。
2. `类别` 是 collection 内部工作类型，如“直接生成”“精化/修复”“泛建模”“多模态邻近”等。
3. `标题` 用论文原始英文标题，不在表格中额外塞入中文译名。
4. `年份` 统一用纯年份数字。
5. `目录` 必须链接到对应单论文目录，而不是只写 slug 文本。
6. 一篇论文在表里只出现一次；不要按“本轮新增”“历史论文”拆成多张正式表。

### 6.7 `数据集与 Benchmark 清单` 的写法

`## 数据集与 Benchmark 清单` 应保持两层结构：

1. 先给 `### 数据集可获取性口径`
2. 再给正式的数据集表

`### 数据集可获取性口径` 默认使用 emoji 对照表，字段至少为：

1. `Emoji`
2. `含义`

正式数据集表的固定列顺序如下：

1. `#`
2. `论文`
3. `评估`
4. `数据集/Benchmark`
5. `来源类型`
6. `制作方法`
7. `输入`
8. `输出`
9. `规模`
10. `可获取性`
11. `获取方式/链接`
12. `简述来源`

表格写法要求如下：

1. `可获取性` 只写 emoji，定义与第 `5.3` 节一致。
2. `获取方式/链接` 只要存在明确 URL，就必须写成 Markdown 链接。
3. 若只有间接入口，例如 GitHub、Zenodo、项目主页、演示视频，也应写出该入口。
4. 若无公开入口，应明确写“原文未提供公开下载链接”或等价表述。
5. 一篇论文可以对应多个数据集/benchmark 条目；此时允许在表中占多行。
6. 若论文没有单独数据集，但有实验案例集、公开工件或标准裁剪样本，也应如实入表，而不是因为“不像标准 benchmark”就漏掉。

### 6.8 `初步归类与覆盖盘点` 的写法

`## 初步归类与覆盖盘点` 默认至少包含以下内容：

1. `### 类别分布`
2. `### BASELINE评估分布`
3. `### 当前最有价值的整体观察`

写法要求如下：

1. `类别分布` 和 `BASELINE评估分布` 优先用三列表格：类别/评估、篇数、说明。
2. `当前最有价值的整体观察` 优先用编号列表，控制在 `3-6` 条。
3. 这里写的是跨论文观察，不重复单篇摘要。
4. 若某个观察直接依赖数据开放性、输入模态、输出模型类型等汇总特征，应确保能从上面的表中追溯出来。

### 6.9 `待补充高优先级候选` 的写法

`## 待补充高优先级候选` 用于记录已经在历史工作中出现、但尚未正式收录的高价值候选。默认表格字段为：

1. `优先级`
2. `标题`
3. `来源文档`
4. `当前价值`

写法要求如下：

1. `来源文档` 应尽量指向当前仓库内已经出现该候选的历史文档来源。
2. `当前价值` 要说明为什么值得补入，例如“可补单步前身工作”“可补 goal modeling 视角”“可补 few-shot completion 对比”。
3. 这里记录的是待正式收录对象，不是随手堆放的大候选池；数量应保持克制。

### 6.10 `更新日志` 的写法

`## 更新日志` 默认使用三列表格：

1. `时间`
2. `更新内容`
3. `说明`

写法要求如下：

1. 时间统一写绝对日期，如 `2026-03-12`。
2. 一行只记录一类相对完整的动作，不要把过细碎的命令级操作逐条入账。
3. 若某轮工作既有新增论文，也有结构调整，可以拆成多行记录。
4. 若一轮检索收获很少或存在失败/阻塞，应在说明列里简明写出原因。

维护要求如下：

1. “当前推荐关键词簇”“高命中特征”“低命中特征”“检索倾向调整”四个小节默认每节不超过 `10` 行。
2. 论文清单必须使用第 5.2 节规定的固定字段，不因每轮新增而换列口径。
3. 数据集与 benchmark 清单必须使用第 5.3 节规定的固定字段。
4. 统计数字必须与当前表格真实内容一致。
5. 已正式入库但尚未完成 `DESC.md` 的论文必须显式标为 `⏳ 尚未提取`。
6. 如果 [SUMMARY.md](./SUMMARY.md) 当前版本已经形成了更细的稳定写法，后续原则上应向当前版本收敛，而不是重新发明结构。

## 7. 工作流程

一轮完整工作默认按以下顺序执行：

1. 先读 [README.md](./README.md)、[GUIDE.md](./GUIDE.md)、[SUMMARY.md](./SUMMARY.md)。
2. 先补历史欠账，优先修复缺失 `DESC.md`、命名不一致、表格未回填等问题。
3. 再进行候选检索与筛选。
4. 对确认收录的论文建立或补齐 `paper.pdf`、`paper_content.txt`、`bibtex.bib`。
5. 按 [DESC_GUIDE.md](./DESC_GUIDE.md) 完成 `DESC.md`。
6. 最后统一回写 [SUMMARY.md](./SUMMARY.md)。

一轮结束前必须做以下一致性检查：

1. 目录中文件是否齐全。
2. `SUMMARY.md` 的论文数与实际目录数是否一致。
3. `BASELINE评估`、类别标签和统计数字是否一致。
4. 数据集表是否与论文总表口径一致。
5. 关键词簇是否仍保持简洁有效，没有变成冗长检索日志。

## 8. 质量与可追溯性要求

后续维护时，以下要求必须满足：

1. 所有结论、数字、标签和分类都应能回溯到原文。
2. 若原文未提供代码、数据集、benchmark 链接，必须如实标注“未提供”，不能臆测补齐。
3. 若某篇论文与 `project_1` 的关系较弱，但仍决定收录，需在 `DESC.md` 和 [SUMMARY.md](./SUMMARY.md) 中明确说明“为何仍保留”。

## 9. 与 `DESC_GUIDE.md` 的关系

只要任务涉及单篇 `DESC.md`，必须再读 [DESC_GUIDE.md](./DESC_GUIDE.md)。

冲突处理规则如下：

1. collection 边界、目录回填和汇总口径，以 [GUIDE.md](./GUIDE.md) 为准。
2. 单篇 `DESC.md` 的章节结构、写法和证据要求，以本文件第 `5.5-5.7` 节和 [DESC_GUIDE.md](./DESC_GUIDE.md) 共同为准。
