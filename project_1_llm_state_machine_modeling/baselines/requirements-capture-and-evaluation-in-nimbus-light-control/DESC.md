#### 1. 案例来源与可获得性
- 论文说明该案例来自 Dagstuhl Seminar 的 Light Control System 挑战问题，作者是在原始问题描述基础上，补足缺失、消解冲突，并形成完整一致的 RSML-e 规格。
- 原文同时说明：完整 LCS 规格和工具信息当时可在 `www.cs.umn.edu/crisys/lcs/` 获取。
- 对当前研究而言，这意味着该论文不仅给出“方法如何建模”，还给出了一个可直接拿来重建控制系统状态机数据样本的经典案例源。

#### 2. 案例对象与系统边界
- 案例对象是建筑内的 **Light Control System (LCS)**，目标是控制房间和走廊中的照明。
- 作者把整个系统看成由多个彼此相对独立的小控制系统组成：每个房间、每段走廊都可以单独建模，再组合成整层楼或整栋楼的控制系统。
- 论文正文主要聚焦“单个房间”的控制问题，因为它已经覆盖了案例里最核心的控制逻辑。
- 单房间系统的核心外部部件包括：
  - 门传感器
  - 光照传感器
  - 运动传感器
  - 设施管理控制台（Facility Manager Console）
  - 两组灯光执行机构：Window Light Bank 与 Wall Light Bank
  - 房间控制面板（Room Control Panel, RCP）
  - 事件日志管理器
- 控制软件本体位于中心，通过单向通信通道与这些部件交互。
- 论文强调：房间模型与走廊模型在功能上相近，但房间案例更完整，因此后续数据集整理优先采用“单房间控制”版本。

#### 3. 单房间案例中最重要的监测量与控制量
原文表 1 给出了作者在“房间级 LCS”中识别出的 monitored / controlled variables。对后续数据集复用，最值得保留的是下面这些变量。

**Monitored variables / 输入侧概念**
- `Light Level`：房间当前照度，范围 `0..10000 lux`
- `Occupied`：房间是否被占用，布尔量
- `Light Level Undetectable`：光照是否不可检测，用于表达光照传感器故障
- `Occupied Undetectable`：占用状态是否不可检测，用于表达门/运动传感器故障
- `Window Light Bank Intensity`：窗侧灯组强度，范围 `0..100`
- `Wall Light Bank Intensity`：墙侧灯组强度，范围 `0..100`

**Operator inputs / 用户输入**
- `Chosen1`、`Chosen2`、`Chosen3`：三个场景按钮
- `Default`：默认场景按钮
- `Set`：设置场景按钮
- `T1`：重新建立默认场景的超时参数，范围 `1..1440` 分钟
- `T3`：关灯超时参数，范围 `1..1440` 分钟
- `FacM Shutoff`：设施管理员远程关灯信号

**Controlled variables / 输出侧概念**
- `ConWindow Light Bank Intensity`：窗侧灯组目标强度，范围 `0..100`
- `ConWall Light Bank Intensity`：墙侧灯组目标强度，范围 `0..100`
- `Failed`：故障指示信号，表示系统检测到组件故障

如果后续要把该案例转成统一状态机数据样本，上述变量已经足够支撑：
- 状态集合抽取
- 事件与输入集合抽取
- 守卫条件抽取
- 动作/输出抽取
- 传感器故障与不可检测模式抽取

#### 4. 房间控制面板（RCP）与用户交互语义
- 原始问题描述中提到 RCP 是一个可移动、独立的设备。
- 作者为了让交互保持简单，采用了类似车载收音机预设键的设计：
  - 用户直接按 `Scene 1/2/3/Default` 选择光照场景
  - 用户先手动调节灯光，再按 `Set + 某个场景键`，用当前房间光照设置覆盖该场景
- 论文明确指出：如果严格按问题描述让“light scene”包含一个文本名字，那么控制面板会复杂很多，但对控制逻辑价值不大，因此作者将其简化为按钮式预设场景。
- 这一点非常关键，因为它说明：
  - 该案例不是纯粹的低层灯光调节
  - 它已经包含了“用户定义模式/场景 -> 系统维护场景”的离散状态控制逻辑

#### 5. Light scene 的完整语义
- 作者认为原题对 light scene 的定义不够合理，因此做了更精细的重构。
- 在作者版本中，一个 light scene 至少包含三类信息：
  - 目标照度（lux）
  - 窗侧灯组强度
  - 墙侧灯组强度
- 这样做的原因是：如果只记录“总照度 + 使用 window/wall/both 的选项”，当用户原本设置的是 `40% window + 60% wall` 时，系统在恢复场景时可能破坏原来的比例，产生不符合直觉的行为。
- 因此，该案例非常适合作为数据源来提炼：
  - `state payload / context variables`
  - “用户设定场景”与“系统维护场景”的关系
  - 连续量（lux）与离散模式（scene）的组合建模

#### 6. REQ 层的关键状态结构
原文图 6 给出的房间级 requirements model 是后续抽取状态机时最有价值的部分之一。其核心状态变量可整理为：

**并行主状态变量**
- `Light_Maintenance_Modes`
- `Chosen_Light_Scene`
- `Failure_Modes`

**层次化子状态**
- `Light_Maintenance_Modes` 下包含：
  - `Room_Occupied`
  - `Room_Empty`
  - `Occupancy_Undetectable`
- `Room_Occupied` 下进一步细分：
  - `Maintain_Light_Scene`
  - `User_Set_Mode`
- `Chosen_Light_Scene` 下包含：
  - `Chosen1_LS`
  - `Chosen2_LS`
  - `Chosen3_LS`
  - `Default_LS`
- `Failure_Modes` 下包含：
  - `Ok`
  - `Failed`

这组状态结构说明该案例不是单一平面 FSM，而是一个典型的**并行 + 层次状态机**源案例，特别适合作为后续 `project_1` 的高价值样本。

#### 7. 关键控制逻辑
作者在 REQ relation 中重点建模了两大任务。

**任务 A：维护用户定义的 light scenes**
- 用户在 `User_Set_Mode` 中调节当前房间灯光。
- 如果用户按下 `Set + Scene_i`，系统把当前照度和两组灯的强度记录为该场景的内容。
- 如果用户直接按某个场景按钮，系统把该场景设置为 current light scene。

**任务 B：在房间占用/空闲条件下维护或关闭灯光**
- 若房间被占用，系统应维持当前场景的照度与灯组比例。
- 若房间空闲，系统需要判断是否：
  - 继续保留当前照明
  - 在 `T3` 到时后关灯
  - 或被设施管理员远程下发 `shutdown` 后立即关灯
- 若房间重新被占用：
  - 若 `T1` 已超时，则恢复到默认场景
  - 否则恢复到此前选择的场景

这一部分对后续数据集提炼尤其重要，因为它能直接支持：
- 状态迁移图重建
- 定时器事件抽取
- 守卫条件与恢复逻辑抽取
- “空闲/重占用/超时/管理员干预”四类典型控制模式抽取

#### 8. 环境模型与仿真方式
Nimbus 并不只是把规格写下来，还让作者为该案例搭建了可执行仿真环境。

**REQ 执行时的输入来源**
- 文本文件
- 用户直接交互输入
- 软件环境模型

**LCS 案例中的具体仿真部件**
- 一个 Microsoft Excel 电子表格，用于模拟：
  - 两组灯的行为
  - 房间照度变化
  - 外部光照变化
  - 房间交通流 / 人员活动等情境
- 一个 Visual Basic 编写的 RCP mockup，用于模拟用户：
  - 设置 window/wall 两组灯强度
  - 输入 `T1` 和 `T3`
  - 选择四个预设场景
  - 触发 `Set` 操作
- 房间占用与设施管理员关灯信号则仍由用户输入提供

对后续数据集建设来说，这意味着该案例不只是“需求文本”，而是天然包含：
- 控制对象
- 传感器
- 执行器
- 操作面板
- 环境扰动
- 仿真接口

这类材料很适合被整理成“需求文本 + 状态机 + 仿真变量字典”的复合样本。

#### 9. 论文中明确暴露出的需求歧义与冲突
作者在仿真和细化过程中发现了不少原始问题中的歧义，其中最值得保留的是两类。

**冲突 1：T1 与 T3 的相互作用**
- 如果用户设置了 `T3 > T1`，那么：
  - 用户离开房间后，灯还未到关灯时间
  - 但当用户重新进入时，系统可能因为 `T1` 已过而把灯切回默认场景
- 这虽然符合原始问题描述，但未必符合用户直觉
- 这是一类典型的“时间约束相互作用导致模式切换语义异常”的案例

**冲突 2：占用检测与故障检测之间的矛盾**
- 论文在把 `Occupied` 从 REQ 细化到 SOFT 时，引入了门传感器和运动传感器
- 同时作者发现如下三条设定会冲突：
  - 如果运动探测器故障，系统要按“被占用”处理
  - 运动传感器覆盖整个房间，能检测细微动作
  - 房门关闭时，房间占用状态不应改变
- 一旦把这些规则同时放进精化模型，就可能在 `Not_Occupied` 与 `Not_Detectable` 条件之间产生重叠，导致模型不一致
- 这对于后续做“生成后验证 / 模型修复”尤其有价值

#### 10. 软件化细化（REQ -> SOFT）中可直接复用的内容
- 作者明确采用四变量模型与 `REQ -> SOFT_REQ -> SOFT` 的细化思想。
- 对 `Occupied` 的细化引入了：
  - 门传感器
  - 运动探测器
  - 上一步状态 `PREV_STEP`
- 这说明该案例可以被拆成两个难度层次：
  - **系统需求层样本**：直接使用 `Occupied`, `Light Level` 等高层 monitored/controlled variables
  - **软件需求层样本**：把高层变量进一步改写为真实传感器与输入接口

如果后续要构建多层数据集，这个案例非常适合做：
- `NL requirements -> abstract state machine`
- `abstract state machine -> sensor/actuator refined state machine`
