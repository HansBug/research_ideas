## 简报

本文关注的问题是：在车辆 MBSE 环境中，如何基于形式化需求自动枚举允许的 operating states 与 transitions，并落到 SysML v2 状态机。

- **输入**：经人工形式化为 LTL 的需求集合
- **方法**：把自然语言需求先翻译成 LTL，再自动枚举可允许状态、筛选可允许迁移，最后编译为 SysML v2 状态机。
- **输出**：SysML v2 状态机中的状态与迁移模型

```text
自然语言需求 -> LTL 形式化 -> 允许状态/迁移生成 -> SysML v2 状态机
```

一句话结论：证明了“需求形式化 + 自动状态/迁移生成 + SysML v2 落盘”的链路可行，适合前期完整性检查。

## 研究问题与动机

### 问题背景
在车辆 MBSE 环境中，如何基于形式化需求自动枚举允许的 operating states 与 transitions，并落到 SysML v2 状态机。

### 核心问题
作者试图回答：在给定 经人工形式化为 LTL 的需求集合 的前提下，怎样更稳定地得到 SysML v2 状态机中的状态与迁移模型，或至少把需求加工为更接近该目标的形式化中间层。

### 研究动机
核心动机是减少需求到行为设计之间的人工鸿沟，提高可追溯性、一致性以及后续验证/执行的可行性。

## 核心方法

### 方法概述
把自然语言需求先翻译成 LTL，再自动枚举可允许状态、筛选可允许迁移，最后编译为 SysML v2 状态机。

### 方法要点
- 任务入口：经人工形式化为 LTL 的需求集合
- 关键处理：LTL 形式化 + 允许状态/迁移自动生成 + SysML v2 编译。
- 最终产物：SysML v2 状态机中的状态与迁移模型（SysML v2 状态机）

## 实验与评估

### 数据集
- **数据/案例**：汽车照明系统示例
- **来源类型**：简化工业示例
- **制作方法**：作者构造示例需求并形式化
- **规模**：1 个示例

### 实验设置
论文用汽车照明系统的简化工业示例逐步展示状态组合、LTL 约束检查与 SysML v2 编译过程。

### 主要实验结果
证明了“需求形式化 + 自动状态/迁移生成 + SysML v2 落盘”的链路可行，适合前期完整性检查。

### 方法的局限性
自然语言到 LTL 的关键一步仍主要依赖人工；示例规模较小；更偏 requirements completion than end-to-end NL modeling。
