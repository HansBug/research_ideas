## 简报

本文关注的问题是：从部分 SysML 模型和 Given-When-Then 需求出发，半自动补全事件驱动关键系统中的状态机迁移。

- **输入**：部分 SysML 模型 + Given-When-Then 需求文本
- **方法**：先将 GWT 需求规整为状态机补全规则，再为已有状态间自动添加 trigger / guard / effect，并补充 traceability 元素。
- **输出**：补全后的 SysML 状态机模型

```text
部分 SysML 状态机 + GWT 需求 -> 规则化补全过程 -> 新增迁移/守卫/动作/追踪关系
```

一句话结论：方法能把事件化需求稳定映射为状态机迁移，并保持需求到模型元素的可追溯性。

## 研究问题与动机

### 问题背景
从部分 SysML 模型和 Given-When-Then 需求出发，半自动补全事件驱动关键系统中的状态机迁移。

### 核心问题
作者试图回答：在给定 部分 SysML 模型 + Given-When-Then 需求文本 的前提下，怎样更稳定地得到 补全后的 SysML 状态机模型，或至少把需求加工为更接近该目标的形式化中间层。

核心动机是减少需求到行为设计之间的人工鸿沟，提高可追溯性、一致性以及后续验证/执行的可行性。

## 实验与评估

### 数据集
- **数据/案例**：ETCS Level 3 + 医疗告警系统案例
- **来源类型**：案例系统
- **制作方法**：正文案例驱动建模
- **规模**：2 个案例

### 实验设置
用 ETCS Level 3 和医疗告警系统两个案例验证可行性，重点展示从需求文本到状态机迁移补全的流程。

### 主要实验结果
方法能把事件化需求稳定映射为状态机迁移，并保持需求到模型元素的可追溯性。

### 方法的局限性
依赖先验存在的状态集合；更像状态机 completion 而不是从零生成；自动化程度仍受需求写法和建模前提影响。
